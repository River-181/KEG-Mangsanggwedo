# 1. 기획

## Q1. 설정한 사용자는 누구이며, 해결하려는 구체적인 문제점/불편함은 무엇인가요?

### 타깃 사용자

**한국의 중소형 학원 원장** (수강생 50~300명 규모, 정직원 강사 3~10명)
- 대표 프로필: 대치동·목동·송파·강남 등 주요 학군의 영어·수학·예체능 학원 원장
- 의사결정 권한은 원장 본인이 단독으로 가지지만, 행정 전담 인력은 없거나 1명 수준

### 해결하려는 구체적 문제점

원장은 매일 **반복적이지만 교육과 무관한 "비교육 업무"를 직접 처리**해야 한다. 아래는 대치동 영어학원 3곳 인터뷰 + 데모 학원 시나리오 기반 추정 빈도:

| 업무 유형 | 추정 빈도 | 현재 처리 방식 |
|---------|--------|--------------|
| 카카오 채널 민원 응대 | 일 10~20건 | 원장이 직접 답변, 평균 응답 6~24시간 |
| 환불·휴원 문의 | 주 3~5건 | 학원법 시행령 수동 확인 + 계산 |
| 결석·보강 조율 | 일 5~10건 | 강사·학부모·학생 3자 조율 |
| 상담 예약 후속 | 주 10~15건 | 상담 기록이 원장 머릿속에만 남아 맥락 소실 |
| 수강료 미납 알림 | 월 20~40건 | 엑셀 수기 확인 후 개별 연락 |

**구조적 불편함 4가지:**
1. **시간 분산**: 교육 품질에 쓸 시간을 행정 업무가 점유 (원장 인터뷰 기준 일 3~4시간 추정)
2. **지연된 응답**: 학부모 만족도와 직결되는 "24시간 내 응답" 기준을 놓치기 쉬움
3. **맥락 소실**: 상담·민원 처리 이력이 원장 개인 기억에 의존 → 인수인계 어려움
4. **법령 오적용 리스크**: 환불 기준(학원법 시행령 제18조)을 사안마다 수동 확인

> 이 문제는 **LMS로는 해결할 수 없는 "학원 운영의 뒷면"**에 있어, AI 기반 운영 에이전트가 필요한 영역이다.

## Q2. 문제를 해결하기 위한 솔루션의 핵심 기능은 무엇인가요?

HagentOS는 **역할 기반 AI 에이전트 팀**이 학원의 비교육 업무를 처리하는 운영 플랫폼이다. Paperclip의 범용 AI 운영 구조를 참고하되, 실제 제품은 한국 학원 원장의 승인 흐름과 k-skill 호출 구조를 중심으로 다시 설계했다.

### ① 4개 역할 에이전트 팀

| 에이전트 | 담당 업무 |
|---------|---------|
| **오케스트레이터 (원장)** | 한 줄 지시 분석 → 적합한 에이전트 자동 배정 |
| **민원담당** | 카카오·전화 민원 분류 + 답변 초안 작성 (학원법·톤가이드 적용) |
| **이탈방어** | 출결 패턴 분석 → 이탈 위험 학생 탐지 + 재등록 개입안 |
| **스케줄러** | 보강·상담·대체 수업 자동 편성 + 구글 캘린더 동기화 |

### ② "한 줄 지시 → 케이스 → 승인 → 발송" 루프

원장이 "이번 주 환불 문의 정리해줘"라고 입력하면:
1. 오케스트레이터가 관련 케이스 수집
2. 민원담당 에이전트가 각 케이스별 답변 초안 작성
3. 원장이 1-click 승인 (혹은 수정 후 승인)
4. 승인된 답변은 카카오·SMS 채널로 전송 (운영 시 채널 provider 토큰 설정 필요)

![[png/01_민원-처리-플로우.png]]
*▲ **민원 처리 플로우** — 학부모 민원 접수부터 AI 답변 초안, 원장 1-click 승인, 카카오 자동 발송까지. k-skill(환불계산기·톤 가이드·법령 조회)이 AI 초안 단계에 자동 주입됨.*

### ③ k-skill 생태계 (한국 교육 특화)

제출 시점 기준으로 한국 학원 업무에 특화된 스킬 패키지를 15종 이상 설계·시드했다. "메시지 템플릿 팩"은 자체 15종 템플릿을 포함한다.

- 환불 계산기 (학원법 시행령 제18조 자동 적용)
- 한국어 톤 가이드 (학부모 상대 존댓말·완곡표현)
- 이탈 위험 계산기 (결석 패턴·성적 하락 복합 스코어링)
- 법령 조회 (law.go.kr MCP 연동)
- 카카오 채널 민원 자동 분류
- 수강료 환불 팩 / 컴플라이언스 팩
- 상담 360° 뷰 / 재등록 플레이북
- 일정 최적화 / 대체 강사 매칭
- 메시지 템플릿 팩 (15종 템플릿 내장)

> 전체 카탈로그는 `skills/hagent/` 및 `skills/community/` 디렉터리에 SKILL.md 표준 형식으로 정리됨.

![[png/05_kskill_생태계_분류도.png]]
*▲ **k-skill 생태계 분류도** — 법령·컴플라이언스, 커뮤니케이션, 리텐션, 스케줄, 분석 5개 범주로 스킬을 묶어 멀티에이전트가 어떤 지식 꾸러미를 호출하는지 보여준다.*

### ④ 채널 연동 (현재 구현 수준)

| 채널 | 상태 |
|------|------|
| 카카오 inbound 파싱 → 케이스 자동 생성 | 구현 완료 (토큰 설정 시 활성) |
| 텔레그램 봇 ↔ 운영자 브리지 | 봇 토큰 기반 polling 구현 |
| SMS (알리고) 발송 | 어댑터 구현, API 키 설정 시 활성 |
| Google Calendar 동기화 | MCP 연동 골격, 토큰 설정 시 활성 |

### ⑤ 자동화 루틴 5종 (DB에 등록된 크론 스케줄)

- 매일 오전 9시 브리핑 (전날 처리 건 + 오늘 우선순위)
- 15분마다 민원 답변 초안 자동 생성
- 주 1회 이탈 위험 학생 리포트
- 매일 오후 8시 다음날 수업 리마인더
- 매월 1일 수강료 미납 알림

> 루틴은 `routines` 테이블에 cron 표현식으로 등록되어 있으며, 운영 환경에서 cron 트리거 연결 시 자동 실행된다.

## Q3. 이 솔루션이 도입되었을 때 기대되는 개선점은 무엇인가요?

### 정량 기대값 (인터뷰·데모 시나리오 기반 추정)

> 아래 수치는 **대치동 학군 영어학원 3곳 인터뷰 + 탄자니아 영어학원 데모 시나리오 기반 추정**이다. 현재 라이브 앱에서 확인되는 것은 케이스·승인·런·루틴 구조이며, 표의 수치는 실운영 KPI가 아니라 도입 효과를 설명하기 위한 추정치다.

| 지표             | 도입 전 (현업 수기 기준) | 도입 후 (기대값)          | 개선 폭        |
| -------------- | -------------- | ------------------- | ----------- |
| 원장 행정 업무 시간    | 일 3~4시간        | 일 30분~1시간 (승인만)     | 약 75% 단축 기대 |
| 학부모 민원 응답 시간   | 평균 6~24시간      | 승인 즉시 자동 발송         | 지연 → 준실시간   |
| 이탈 위험 학생 탐지 시점 | 학부모가 환불 요청 시   | 주 1회 자동 리포트로 사전 탐지  | 사후 → 사전     |
| 환불 계산 정확도      | 원장 기억·수기 참조    | 학원법 시행령 제18조 자동 적용  | 일관성 확보      |
| 운영 의사결정 기록     | 원장 머릿속에만 존재    | 모든 케이스·런·승인 DB 영속화  | 전수 추적 가능    |

### 정성 개선 (운영 설계 관점)

1. **수평 확장 용이**: 지점 추가 시 에이전트 설정·k-skill을 복제해 동일 운영 규격 유지
2. **원장 집중도 회복**: 반복 행정에서 자유로워져 교육 기획·강사 관리에 시간 배분
3. **학부모 경험 품질**: 답변 톤 일관성 + 응답 지연 감소
4. **규정 준수 가시화**: 환불·상담·개인정보 처리 전 과정이 DB 기록으로 남음

### 확장 방향

- **다채널**: 텔레그램·디스코드·슬랙 등 추가 (현재 카카오·텔레그램 연동 골격 구현)
- **다업종**: 운영 페인이 유사한 소규모 SMB (학원 외 소규모 교습소·상담센터 등)로 일반화 여지
- **데이터 축적 효과**: 의사결정 로그가 축적될수록 에이전트 제안 품질을 개선할 수 있는 근거 데이터가 쌓임

## Q4. 이 프로젝트에서 사용할 AI 도구(Claude Code, Cursor, Codex 등)와 모델은 무엇이며, 선택한 이유는 무엇인가요?

### 도구별 운영 전략·제약

**Claude Code (Opus 4.6 / Sonnet 4.6)** — 주 구현 도구
- 핵심 전략: Plan Mode로 먼저 설계 → 병렬 서브에이전트(Explore·Plan·code-reviewer)로 탐색/리뷰 분산 → 최종 변경은 메인 컨텍스트에서 단일 커밋
- 제약: 대용량 파일은 offset/limit 파라미터로 부분 로드하여 토큰 초과 방지
- 적용 예: 7일간 서버·UI·DB 마이그레이션·skill runtime을 한 에이전트가 일관되게 구현

**Codex CLI (GPT-5 / GPT-5.4)** — 2차 리뷰어
- 핵심 전략: Claude 산출물에 대한 독립적 감사. 코드 변경 권한은 주지 않고 감사 리포트만 수신
- 제약: 터미널 기반이라 장기 대화 상태 관리가 Claude보다 제한적
- 적용 예: Paperclip 레퍼런스 감사 후 HagentOS 도메인 재해석 제안

**ChatGPT Web (GPT-5.4)** — 전략·기획
- 핵심 전략: Canvas로 긴 구조화 문서 초안 작성 → Obsidian 볼트에 옮겨 정제
- 제약: 웹 기반이라 로컬 파일 직접 접근 불가. 복붙 핸드오프로 통합
- 적용 예: 대회 플레이북 작성, 심사 기준별 자기 평가

**Perplexity / Gemini Deep Research** — 리서치
- 핵심 전략: "검색+합성"이 필요한 질문에만 할당 (기관·경쟁·법령·통계)
- 제약: 최신 웹 편향 — 오래된 내부 자료나 비공개 문서는 검증 불가
- 적용 예: 코리아교육그룹 계보 분석, 학원법 관련 판례 조사

## Q6. 토큰 낭비를 최소화하고 유지보수성 및 재현성을 높이기 위한 전략이 있다면 작성해주세요.

### 토큰 절감 전략 (5가지)

**① 서브에이전트 분리 — 요약본만 수신**
- 패턴: 메인 세션은 Explore 서브에이전트에게 탐색을 맡기고, 요약 결과만 받음
- 효과: 탐색 과정의 수만~수십만 토큰을 메인 컨텍스트에서 배제

**② CLAUDE.md 규칙으로 반복 프롬프트 제거**
- `.claude/CLAUDE.md`에 프로젝트 규칙·구조 지도를 기록
- 매 세션마다 "우리 스택이 뭐고 어디에 뭐가 있는지" 재설명 불필요

**③ Context compaction**
- 긴 대화에서 Claude Code의 자동 컴팩션 기능 활용
- 중요 결정·코드 스니펫은 `decision-log.md`·git commit에 외재화

**④ 도구 특성에 맞는 분업**
- "이 질문은 누가 답하면 가장 토큰 효율이 좋은가"를 먼저 판단
- 출처 필요 질문은 Perplexity, 긴 생성은 Claude Opus, 빠른 판단은 Haiku

**⑤ 파일 부분 읽기**
- Read tool의 offset/limit 파라미터로 대용량 파일도 필요 구간만 로드

**누적 기록**: 7일간 33회 이상 주요 세션 + 누적 추정 토큰 사용량은 `ai-usage-stats.md` 참조. 서브에이전트 분리·컴팩션·외재화 조합이 없었다면 동일 결과물을 만드는 데 더 큰 토큰 비용이 들었을 가능성이 높다.

### 유지보수성 전략 (4가지)

**① 계약 기반 문서 (`.agent/system/contracts/`)**
- `workspace-contract.md`·`memory-evidence-policy.md`로 워크스페이스 운영 규칙 명시
- 새 AI나 사람이 들어와도 계약만 읽으면 즉시 동기화

**② 스킬 표준 형식 (SKILL.md)**
- 모든 k-skill을 동일한 YAML frontmatter + 섹션 구조로 통일
- `.agent/skills/` 아래 10+ 개 스킬이 같은 형식 → 추가·수정 용이

**③ 의사결정 로그 (`decision-log.md`)**
- 모든 아키텍처 결정에 DEC-NNN 번호 부여 + 근거·대안·결과 기록
- 후속 AI가 과거 결정의 배경을 이해하고 일관성 유지

**④ 단일 진실 소스 (PROGRESS.md)**
- 멀티 AI 동시 작업 시 중복·충돌 방지
- 모든 AI가 작업 시작 전 PROGRESS 확인 의무

### 재현성 전략 (4가지)

**① AI 세션 intake CSV 체계**
- 매 세션 종료 시 `ai-session-intake.csv`에 row append (필수 필드: 세션ID·도구·모델·토큰·artifact)

- **현재 33+ 세션이 이 형식으로 기록됨**

**② 프롬프트 카탈로그 (`prompt-catalog.md`)**
- 재사용 가능한 고품질 프롬프트를 목적별로 분류 보관
