## 1. 프로젝트 개요

### 해결한 문제

한국의 중소형 학원 원장은 하루 평균 20~40건의 비교육 업무(민원, 환불 요청, 결석 처리, 보강 일정 조율, 상담 후속 조치)를 직접 처리한다. 이 업무들은 반복적이고 패턴이 명확하지만, 전담 인력을 두기엔 규모가 작고, 방치하면 재원율과 학부모 만족도가 직결된다.

**HagentOS**는 이 비교육 업무를 역할 기반 AI 에이전트 팀이 처리하는 학원 운영 플랫폼이다. 원장은 "이번 주 환불 민원 처리해줘" 한 줄로 지시하면, 오케스트레이터가 민원담당 에이전트에게 케이스를 배정하고, 답변 초안을 작성해 승인 대기 상태로 올린다.

### 핵심 기능

| 기능 | 설명 |
|------|------|
| 케이스 관리 | 민원·상담·법률질문을 케이스로 분류하고 에이전트에 자동 배정 |
| 멀티에이전트 팀 | 오케스트레이터·민원담당·이탈방어·스케줄러 4종 에이전트 |
| 승인 게이트 | AI 초안 → 원장 1-click 승인 → 발송 흐름 |
| k-skill 생태계 | 한국 교육 현장 특화 스킬 패키지 (환불규정·톤가이드·이탈분석 등) |
| 채널 연동 | 카카오·텔레그램 inbound 파싱 → 케이스 자동 생성 |
| 자동화 루틴 | 매일 9시 브리핑, 주 1회 이탈 위험 리포트 등 5종 |

### 데모

- **라이브 URL**: https://hagent-os.up.railway.app
- **GitHub**: https://github.com/River-181/hagent-os
- **데모 학원**: Tanzania English Academy (탄자니아 영어학원, 대치동 컨셉)
- **기술 스택**: React 19 + Vite + TypeScript / Express + TypeScript / Neon PostgreSQL / Railway 배포

## 2. AI 도구 활용 전략

### 사용 도구 개요

| 도구 | 용도 | 세션 수 | 추정 토큰 |
|------|------|---------|---------|
| Claude Code (Opus 4.6 / Sonnet 4.6) | 주 개발 도구 — 설계·구현·디버깅·테스트 | 15+ | ~1.2M |
| Codex (GPT-5 / GPT-5.4) | 코드 감사·구조 설계·문서 작성·증빙 운영 | 13+ | ~198K |
| ChatGPT Web (GPT-5.4) | 대회 전략·제품 방향·프롬프트 정제 | 4+ | ~70K |
| Perplexity | 리서치 — 기관 분석·경쟁 분석·교육 시장 | 2+ | ~18K |
| NotebookLM | 리서치 합성 | 1 | - |

> 총 33개 이상 AI 세션 공식 기록. `04_증빙/01_핵심로그/ai-usage-log.md` 참조.

### 프롬프트 전략 3단계

**탐색 단계 (Day 1-2)**
- "500팀이 참가하는 대회에서 LMS 클론을 피하고 한국 교육 운영자의 페인을 날카롭게 찌를 아이디어를 구체적인 시나리오와 함께 5개 제안해줘"
- "paperclip 오픈소스 아키텍처를 한국 학원 운영에 맞게 재해석한다면 어떤 도메인 모델이 필요한가"

**구현 단계 (Day 3-6)**
- 명세 기반 프롬프트: "아래 API 스펙을 구현해줘. 기존 케이스 라우터 패턴과 일관성 유지, Drizzle ORM 사용, TypeScript strict"
- 회귀 테스트 체크리스트 자동 생성: "이 커밋이 영향을 미칠 수 있는 사이드 이펙트 10개를 열거하고 각각 테스트 방법을 말해줘"
- 버그 진단: "이 에러 로그와 관련 코드 3개 파일을 주면 근본 원인과 수정 방법을 단계별로 설명해줘"

**마감 단계 (Day 7)**
- 재평가 루프: "KEG 심사 기준 4개 축으로 현재 제품을 90점 기준으로 평가하고, 90→92점으로 올릴 퀵윈 3개만 구체적으로 말해줘"
- 전수 감사: "배포된 앱의 모든 페이지와 API 엔드포인트를 체계적으로 테스트하고 broken 항목을 리스트로 만들어줘"

### AI 협업 구조

8개 역할 기반 에이전트를 구성해 업무를 분산했다:

| 에이전트 역할 | 담당 도구 | 주요 아웃풋 |
|-------------|---------|------------|
| PM | ChatGPT GPT-5.4 | 대회 전략, 제품 방향, 스코프 결정 |
| Research | Perplexity | 기관 분석, 시장 리서치, 경쟁 분석 |
| Product | ChatGPT / Claude | PRD, 기능 정의, 사용자 시나리오 |
| Builder (주) | Claude Code Opus/Sonnet | 전체 코드베이스 구현·리팩터 |
| Reviewer | Codex GPT-5.4 | 코드 감사, 구조 검토, 회귀 테스트 |
| QA | Claude Code (서브에이전트) | 엔드포인트 테스트, 배포 검증 |
| Evidence | Codex GPT-5 | intake CSV 작성, 증빙 정본 관리 |
| Submission | Claude Code | AI 리포트, README, 체크리스트 |

이 구조는 Paperclip식 멀티에이전트 운영을 참고하되, 실제 학원 업무에 맞게 k-skill 호출과 제출 증빙 역할까지 포함하도록 확장한 운영 체계다.

핸드오프는 `.agent/AGENTS.md` 공용 진입점과 `.agent/system/ops/PROGRESS.md` 단일 진실 소스로 표준화했다.

## 3. 핵심 문제 해결 과정

### Day 1-2: 방향 설정

ChatGPT와 Perplexity를 사용해 대회 맥락과 교육 시장을 분석했다. "LMS 금지"라는 심사 기준을 역으로 활용해 **운영 행정 자동화**로 방향을 잡았다. Paperclip(일반용 AI 운영 플랫폼)을 참고 아키텍처로 삼되, 한국 학원 운영 도메인에 특화된 컨트롤 플레인으로 재해석하기로 결정했다.

핵심 의사결정: "케이스 → 에이전트 → 승인 → 문서" 루프를 제품의 핵심 플로우로 확정.

### Day 3-5: 핵심 구현

Claude Code로 독립 저장소를 구축하고 E2E를 완전 재구현했다. 주요 구현 난제:

1. **멀티에이전트 orchestration**: 오케스트레이터가 케이스를 분석해 적합한 역할 에이전트에게 자동 배정하는 로직. Claude Code가 drizzle-orm 기반 DB 조회와 role-matching 알고리즘을 함께 구현했다.

2. **k-skill 런타임**: 에이전트별로 설치·마운트·활성화 상태를 관리하는 스킬 생태계. "YAML 기반 스킬 메타데이터 + DB 레지스트리 + 런타임 컨텍스트 주입" 구조를 Claude Code가 설계하고 구현했다.

3. **채널 파서**: 카카오 inbound 메시지를 파싱해 케이스 타입을 자동 분류하는 로직. Claude가 한국어 자연어 패턴을 기반으로 분류 규칙을 직접 제안하고 코드화했다.

### Day 6: 보강 및 데모 시딩

시드 데이터 품질이 심사 인상에 직결된다고 판단해 탄자니아 영어학원 데모를 정교화했다. 법령 일정이 매주 반복되는 비현실적 데이터를 수정하고, 종일 이벤트를 구글 캘린더 스타일 상단 고정 행으로 표시하는 UX 개선을 진행했다.

### Day 7: 전수 검증 + 마감

Claude Code 서브에이전트 5개를 병렬로 실행해 전체 페이지·버튼·API 엔드포인트를 체계적으로 테스트했다. 발견된 문제:
- OpenAI API를 실존하지 않는 모델(gpt-5-codex)로 호출하고 있던 silent bug → Chat Completions API + gpt-4o-mini로 교체
- DB 에이전트 레코드의 model 필드가 구버전 값으로 남아있던 문제 → 기동 시 자동 패치 마이그레이션 추가
- 루틴 백필 로직의 조건 분기 버그 → 독립 체크로 수정

## 4. AI가 한 일 vs 사람이 한 일

### AI가 주도한 영역

- **코드 생성**: 전체 코드베이스의 90% 이상을 Claude Code가 초안 작성
- **버그 탐지 및 수정**: 에러 로그 + 관련 파일 제공 시 근본 원인 진단과 수정 코드까지 제안
- **회귀 테스트**: 각 커밋마다 사이드 이펙트 체크리스트 자동 생성
- **데모 시드 데이터**: 학생 48명, 강사 6명, 학부모, 일정, 케이스 등 현실적 더미 데이터 생성
- **문서 초안**: README, AGENTS.md, handoff 문서, AI 리포트 초안
- **증빙 관리**: 33개 세션 기록, intake CSV 작성, 원장 업데이트

### 사람이 결정한 영역

- **문제 선택과 스코프**: "민원 자동화"라는 방향과 타깃 사용자(학원 원장) 확정
- **아키텍처 철학**: "케이스-승인-문서 루프"와 "k-skill 생태계" 개념 정의
- **우선순위 판단**: 7일이라는 제한에서 무엇을 만들고 무엇을 버릴지
- **데모 시나리오**: 심사위원이 2분 안에 핵심 가치를 체험할 시연 흐름 설계
- **품질 게이트**: AI 초안을 검토하고 "이건 다시" / "이건 맞다" 판단
- **최종 제출 결정**: 무엇을 제출물로 확정할지

### 한계

- **OpenAI quota**: 현재 배포 환경에서 OpenAI API quota 초과로 실시간 AI 응답은 크레딧 충전 필요
- **채널 실증**: 카카오·텔레그램 실제 메시지 delivery는 prod 토큰 없이 데모 모드로 동작
- **모바일 반응형**: 기본 반응형은 구현됐으나 모바일 특화 최적화는 미완
- **법령 실시간 조회**: LAW_GO_KR_OC 미설정 시 내장 발췌본 fallback으로 동작

## 6. AI 협업의 핵심 교훈

**"AI는 빠른 손이고, 사람은 방향타다"**

7일 개발에서 코드 자체는 AI가 썼지만, 어떤 코드를 쓸지 결정하는 의사결정은 일관되게 사람이 했다. AI가 제안한 아키텍처도 "이게 학원 원장의 실제 워크플로우와 맞는가"를 인간이 검토하지 않으면 그럴듯하지만 쓸모없는 코드가 된다.

멀티에이전트 협업 구조(8개 역할 분리, PROGRESS.md 단일 진실 소스, 핸드오프 계약)는 단순히 "여러 AI를 쓴 것"이 아니라 팀처럼 AI를 운영하는 방법론이었다. 여기에 k-skill 생태계를 얹어 도메인 지식을 재사용 가능하게 만든 점이 7일이라는 제약 안에서 완성도를 끌어올린 핵심 레버리지였다.
