# Bottom-Up Academy Research Source

> `spec`과 사용자 플로우로 가기 전에, 한국 학원 생태계를 바닥부터 조사하기 위한 기준 문서.
> 이 문서는 “무슨 기능을 만들까”보다 “현장에서 어떤 운영 마찰이 반복되는가”를 먼저 보게 만든다.

## 리서치 기본 원칙

- 탑다운으로 큰 시장 이야기부터 누르지 않는다.
- 현장의 작은 반복 업무, 감정 노동, 수기 운영, 예외 처리부터 본다.
- `운영자`, `실무자`, `학부모`, `수강생`의 하루를 관찰하는 방식으로 접근한다.
- 기능이 아니라 `업무`, `병목`, `예외`, `승인`, `기록`, `책임소재`를 먼저 수집한다.

## 바텀업으로 봐야 하는 이유

- 학원은 겉으로는 “출결·수납·상담”으로 보이지만, 실제 운영은 그보다 훨씬 자잘한 예외들로 돌아간다.
- 예:
  - 밤 늦은 학부모 민원
  - 강사 보강/대타/휴강 조정
  - 재등록 직전 불안 신호
  - 학생별 특이사항 누적
  - 차량/시설/법무/세금/노무 같은 운영 꼬리업무
- 이 자잘한 운영 마찰이 제대로 드러나야 `EduPaperclip`이 단순 챗봇이 아니라 운영 시스템으로 설득력을 가진다.

## 조사 대상 층위

### 1. 사람 층위

- 원장 / 부원장 / 실장 / 상담 직원
- 강사 / 담임 / 조교 / 행정 담당
- 학부모
- 수강생

### 2. 운영 층위

- 신규 문의와 등록
- 반 편성 / 시간표 / 보강 / 결석 / 이탈 조짐
- 학부모 응대 / 민원 / 상담 기록
- 강사 스케줄 / 피로 / 성과 / 대체 배정
- 보고 / 회계 / 세금 / 법무 / 차량 / 시설

### 3. 도메인 층위

- 보습 / 입시
- IT 아카데미 / 부트캠프
- 태권도 / 예체능
- 운전면허
- 기타 레슨형 교육

## 지금 리서치에서 꼭 뽑아야 하는 것

1. 하루 동안 반복되는 운영 업무
2. 사람이 꼭 끼어드는 승인 지점
3. 감정 노동이 발생하는 순간
4. 정보가 흩어져 있어서 다시 묻고 다시 찾는 순간
5. 돈이 새거나 이탈이 발생하는 순간
6. 기존 솔루션이 건드리지 않는 빈칸

## NotebookLM이 잘해야 하는 일

- 여러 raw source를 읽고 `운영 흐름`을 합성한다.
- 표면적인 카테고리보다 `현장 단위의 반복 업무`를 추출한다.
- 서로 다른 출처에서 반복적으로 나타나는 pain을 묶는다.
- 아직 증거가 약한 것은 가설로 남기고, 강한 것은 후보로 올린다.

## NotebookLM이 하면 안 되는 일

- “AI를 넣으면 다 좋아진다” 식으로 포장하기
- 기능 리스트를 먼저 제안하기
- 공교육/사교육/도메인별 차이를 뭉개기
- 한두 개 사례를 전체 시장의 진실처럼 일반화하기

## 이 조사로 최종적으로 얻고 싶은 것

- 첫 번째로 집중할 사용자 1개
- 그 사용자가 가장 자주 겪는 운영 pain 1개
- 그 pain을 설명하는 하루 흐름
- 기존 대안의 한계
- 왜 `role-based AI team`이 필요한지에 대한 근거
