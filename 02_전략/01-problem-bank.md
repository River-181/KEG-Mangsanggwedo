## 경쟁 현황 요약 (개발 전 필수 인식)

**기존 솔루션이 이미 하는 것:**
- 출결·수납·학부모 알림 자동화: 클래스업, 온하이, 랠리즈 등 (학원)
- 생활기록부 AI 초안: LGU+ U+슈퍼스쿨, 왓퀴즈, AI생기부 앱 (공교육)

**아무도 안 하는 것 (화이트스페이스):**
- 역할 기반 AI 에이전트 팀 오케스트레이션 (여러 업무를 한 번의 지시로 분담 처리)
- 신규 상담 → 등록 → 이탈 예측 전 과정 인텔리전트 처리
- 학부모 비정형 민원의 맥락 파악 + 응답 초안 + 에스컬레이션 판단
- 강사 성과/피로도 자동 분석

## P-001. 학원 원장 운영 보조 Swarm

| 항목 | 내용 |
|------|------|
| **사용자** | 사교육 / 운영자 (보습·IT·예체능 학원 원장) |
| **문제 문장** | 학원 원장은 신규 상담, 학부모 민원, 강사 관리, 이탈 예측을 혼자 처리하며 하루 3시간 행정에 월 300만원 기회비용을 낭비한다 |
| **현재 방식** | 카카오톡 + 전화 + 수기 스케줄표. 출결/수납은 클래스업 등 사용 |
| **병목** | ① 신규 상담 문의: 사람이 직접 응대 ② 학부모 비정형 민원: 야간까지 원장 직접 처리 ③ 재원생 이탈: 징후 모르다가 결제 안 하고 나감 ④ 강사 관리: 스케줄·워크로드·피드백 구두로만 처리 |
| **기존 솔루션 한계** | 클래스업·온하이 등은 출결·수납만. AI 없음. 민원·상담·이탈 미지원 |
| **AI swarm이 필요한 이유** | 민원 분류 agent + 상담 응대 agent + 이탈 예측 agent가 병렬 처리. 단일 챗봇으로는 맥락 유지 어려움 |
| **MVP 핵심 기능** | ① 학부모 민원 분류 + 응답 초안 ② 재원생 출결 이상 감지 + 이탈 경보 ③ 원장 한 줄 지시 → 2~3개 업무 분담 처리 |
| **2분 데모 장면** | 원장이 "오늘 민원 처리하고 이번 주 이탈 위험 학생 알려줘" 입력 → agent A: 민원 3건 초안, agent B: 출결 이상 학생 목록 |
| **현장 근거** | 🟢 bati.ai: 원장 행정 하루 3시간 = 월 300만원 손실 분석 🟡 클래스업/온하이 기능 목록: 출결·수납만 커버, 상담·이탈은 없음 |
| **경쟁 위협** | 없음 (역할 기반 swarm 구조의 학원 솔루션 미존재) |
| **리스크** | 실제 학원 데이터 없이 mock만으로 데모 설득력 낮아질 수 있음 |

## P-002. 담임교사 반 운영 Swarm

| 항목 | 내용 |
|------|------|
| **사용자** | 공교육 / 실무자 (초중고 담임교사) |
| **문제 문장** | 한국 담임교사는 행정업무 주 6시간(OECD 1위)에 학부모 민원 스트레스 56.9%(세계 2위)를 안고, 수업보다 행정에 더 많은 시간을 쓴다 |
| **현재 방식** | NEIS + 카카오톡 + 개인 메모. 생기부는 직접 작성 |
| **병목** | ① 학부모 민원: 야간·주말 포함 비정형 응대 ② 반 행정: 공문·가정통신문 누적 ③ 생기부: 학생 30명 × 1~2시간 ④ 수업 계획: 개인화 피드백 제공 시간 없음 |
| **기존 솔루션 한계** | LGU+ 슈퍼스쿨: 생기부 자동화 + 출결관리 (B2G 납품). 왓퀴즈: 생기부 초안. **민원 지능형 처리·반 행정 통합·역할 기반 swarm 없음** |
| **AI swarm이 필요한 이유** | 민원 agent + 행정처리 agent + 학생프로필 agent가 담임의 한 줄 지시에 반응 |
| **MVP 핵심 기능** | ① 학부모 민원 분류 + 응답 초안 ② 수업 계획 초안 자동 생성 ③ 학생별 상황 요약 (출결·특이사항 기반) |
| **2분 데모 장면** | 교사가 "이번 주 민원 처리 + 목요일 수업 계획" 입력 → agent A: 민원 2건 초안, agent B: 단원별 수업 계획 출력 |
| **현장 근거** | 🟢 OECD TALIS 2024: 교사 행정업무 주 6시간 (OECD 평균의 2배, 세계 1위) 🟢 학부모 민원 스트레스 56.9% (세계 2위) 🟡 수업 시간이 OECD보다 4시간 적음 |
| **경쟁 위협** | 생기부 단독은 경쟁 있음 (LGU+, 왓퀴즈). 역할 기반 swarm은 없음 |
| **리스크** | 공교육 개인정보 규제 (NEIS 연동 불가). mock 학생 데이터로 데모 구성 필요 |

## P-003. 학원 신규 상담 자동화 (신규 발굴)

| 항목 | 내용 |
|------|------|
| **사용자** | 사교육 / 운영자 (학원 원장 또는 상담 담당자) |
| **문제 문장** | 신규 수강생 유입은 문의 → 상담예약 → 레벨테스트 → 등록 → 반배정 전 과정이 사람 손에 달려 있어 문의 응대 지연이 등록 포기로 이어진다 |
| **현재 방식** | 전화/카카오톡 문의 → 원장·상담 담당이 직접 처리 |
| **병목** | 야간·주말 문의 미응대. 상담 예약 누락. 레벨 진단 전 과정 수작업 |
| **기존 솔루션 한계** | 클래스업 등은 등록 이후 관리만. 신규 유입 전 과정 미지원 |
| **AI swarm이 필요한 이유** | 문의 응대 agent + 레벨 진단 agent + 반 추천 agent가 24시간 처리 |
| **MVP 핵심 기능** | ① 자동 문의 응답 + 상담 예약 안내 ② 간단 레벨 진단 ③ 적합 반 추천 출력 |
| **2분 데모 장면** | 학부모가 "중2 수학 문의" → swarm이 레벨 진단 → "B반 추천" 자동 안내 |
| **현장 근거** | 🟡 bati.ai: 신규 상담 프로세스 자동화가 관리 프로그램의 다음 단계로 명시 🔴 전환율 데이터 미확보 |
| **경쟁 위협** | 없음 |
| **리스크** | 카카오채널 연동 구현 난이도. 7일 MVP 범위 조정 필요 |
| **포지셔닝** | P-001의 확장 기능 또는 데모의 한 장면으로 포함 가능 |

## P-004. 학부모 민원 지능형 처리 (Cross-cutting)

| 항목 | 내용 |
|------|------|
| **사용자** | 공교육 담임교사 + 사교육 원장/강사 공통 |
| **문제 문장** | 학부모 민원은 시간·내용·긴급도가 예측 불가한 비정형으로 들어오며, 응대자가 퇴근 후에도 직접 처리해야 한다 |
| **병목** | 비정형 맥락 파악 실패 → 재응대 → 감정 노동 누적 |
| **AI 필요 이유** | 민원 분류 + 맥락 파악 + 응답 초안 + 에스컬레이션 판단을 agent가 처리 |
| **현장 근거** | 🟢 교사 민원 스트레스 56.9% (세계 2위, OECD TALIS 2024) 🟡 학원 민원이 원장 행정 3시간의 일부 (bati.ai) |
| **포지셔닝** | P-001 또는 P-002에 통합하는 핵심 기능. 독립 제품 불가 |

## 현재 권장 방향

```
MVP 1순위: P-001 학원 원장 운영 보조 Swarm
  → KEG 주최사 직접 공감. 현장 데이터 근거(월 300만원) 있음.
     기존 솔루션 명확한 공백. 역할 기반 swarm이 차별점.

MVP 2순위: P-002 담임교사 반 운영 Swarm
  → OECD 수치로 현장성 강함. 데모가 직관적.
     생기부는 경쟁 있으므로 민원+수업계획 중심으로 재포커스.

확장 기능: P-003 신규 상담 자동화
  → P-001에 포함하거나 데모의 한 장면으로 제시

핵심 차별점 (두 후보 공통):
  "단순 도구가 아니라 역할 기반 AI 팀이 분담 처리한다"
```
