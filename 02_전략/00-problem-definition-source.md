# Problem Definition Source

> 기능을 정하기 전에 읽는 기준 문서.
> 이 문서는 `spec`이 아니라, 문제 정의와 페르소나를 세우기 위한 출발점이다.

## 이 문서의 목적

- 아직 기능을 확정하지 않는다.
- 먼저 `누가`, `어떤 상황에서`, `무엇 때문에`, `왜 지금`, `기존 방식으로 왜 안 되는가`를 정리한다.
- Claude나 다른 AI가 너무 빨리 기능 목록, 화면, 아키텍처로 내려가지 않도록 기준을 고정한다.

## 현재 프로젝트의 상위 방향

- 상위 아이디어: `EduPaperclip`
- 정의: `paperclip`의 역할 기반 AI orchestration 개념을 한국 교육 현실에 맞게 변형한 `K-교육 제로 휴먼 운영 시스템`
- 핵심 차별점: 단일 AI 도구가 아니라, 역할을 나눈 AI 팀이 운영 업무를 분담 처리한다.

## 지금 확정된 것

- 프로젝트는 `운영 시스템` 관점으로 간다.
- 공교육/사교육 모두 확장 가능성을 보지만, MVP는 한 역할군에 집중해야 한다.
- 지금 단계의 문제 정의는 `세부 기능 확정`이 아니라 `운영자 관점 시스템 vs 강사/교사 관점 시스템` 중 어떤 축을 먼저 풀 것인지 정하는 것이다.

## 아직 확정되지 않은 것

- 최우선 사용자:
  - 사교육 운영자
  - 공교육/사교육의 교사·강사
- 첫 MVP의 대표 시나리오
- 첫 데모에서 보여줄 핵심 업무 1~3개
- agent orchestration을 어느 정도까지 실제 동작으로 보여줄지

## 문제 공간 요약

### 공통 문제

- 교육 현장에는 반복적이고 감정 소모가 큰 운영 잡무가 많다.
- 대표 범주:
  - 학부모 민원
  - 학생 관리
  - 수업/행정 준비
  - 일정/운영 조율
  - 상담/기록 정리

### 사교육 운영자 관점

- 원장은 신규 상담, 민원, 강사 관리, 재등록, 이탈 대응, 운영 이슈를 동시에 떠안는다.
- 기존 솔루션은 출결/수납/알림은 다루지만, 비정형 민원/상담/이탈/강사 운영은 거의 비어 있다.
- 강한 가설:
  - “학원 운영자가 수업과 의사결정에만 집중하고, 나머지 운영 잡무는 AI 팀이 처리하게 할 수 있는가?”

### 교사/강사 관점

- 담임/강사는 수업보다 행정과 민원, 기록 업무에 많은 시간을 쓴다.
- 단순 생성형 도구보다 `민원`, `수업 준비`, `학생 요약`, `상담 초안`을 나눠 처리하는 역할 기반 시스템이 더 적합할 수 있다.
- 강한 가설:
  - “교사와 강사의 업무를 보조하는 개인 비서 수준을 넘어서, 반/수업 운영팀처럼 작동하는 AI 팀을 만들 수 있는가?”

## 경쟁 지형에서 중요한 점

- 레드오션:
  - 출결
  - 수납
  - 알림 자동화
  - 생기부 단독 초안
- 화이트스페이스:
  - 역할 기반 AI 에이전트 팀 오케스트레이션
  - 비정형 민원 처리 + 응답 초안 + 판단 보조
  - 상담/이탈/강사 관리 같은 운영 맥락 통합
  - 교육 운영 데이터의 자산화

## 현재 살아 있는 문제 후보

### 후보 A. 학원 원장 운영 보조

- 강점:
  - KEG 주최 맥락과 직접 맞닿음
  - 운영/실무 접합성이 강함
  - 확장성, 사업성, 데이터 자산화 스토리가 좋음
- 약점:
  - 범위가 쉽게 커진다
  - mock 데이터가 약하면 설득력이 떨어질 수 있다

### 후보 B. 담임/교사 반 운영 보조

- 강점:
  - 공신력 있는 데이터가 강함
  - 데모가 더 직관적일 수 있다
  - 외부 연동 없이도 시뮬레이션이 쉬움
- 약점:
  - 경쟁 서비스와 겹쳐 보일 위험이 있다
  - 운영 시스템보다 개인 도구처럼 보일 수 있다

## 현재 판단

- 지금은 `후보 A vs 후보 B`를 가르는 단계이지, 구체 기능을 확정하는 단계가 아니다.
- 기능 문서는 이 판단이 끝난 뒤에 작성한다.
- 따라서 지금 Claude가 해야 할 일은:
  - 기능 정의
  - 화면 정의
  - 상세 아키텍처
  - API 설계
  가 아니라,
  - 문제 정의 초안
  - 핵심 페르소나 1~2개
  - jobs-to-be-done
  - 현재 대안과 한계
  - 왜 AI 팀이어야 하는지
  를 정리하는 것이다.

## 기능 전에 먼저 만들어야 할 기반자료

1. 문제 정의 초안
2. 대표 페르소나 1~2개
3. 각 페르소나의 하루 흐름과 병목
4. 현재 대안과 그 한계
5. 왜 단일 챗봇이 아니라 역할 기반 AI 팀이어야 하는지
6. 지금 풀 문제와 나중에 미룰 문제의 경계

## Claude가 하면 안 되는 것

- 기능 목록부터 길게 쓰기
- 화면 구성부터 상상해서 문서 채우기
- 모든 사용자군을 한 번에 만족시키려 하기
- `좋아 보이는 기능`을 문제 정의보다 앞세우기
- 문제/페르소나/기존 대안 분석 없이 바로 `spec`으로 내려가기
