# 데이터 자산화 가능 영역

## 한 줄 요약

학원 운영 데이터는 이미 많다. 문제는 흩어져 있고, `케이스`, `승인`, `후속 행동`, `성과`와 연결되지 않아 자산이 되지 못한다는 점이다.

## 자산화 가능한 데이터 축

### 1. 학생 데이터

- 기본 인적 정보, 학년, 반, 등록/휴원/퇴원 이력
- 출결 패턴, 지각/결석 사유, 보강 이력
- 성적/평가/오답/학습 진도
- 상담 메모, 행동 특이사항, 이탈 징후

### 2. 학부모 데이터

- 문의/민원/상담 채널과 이력
- 반응 속도, 관심사, 민감 포인트, 약속 이력
- 결제 패턴, 환불/할인 요청, 재등록 확률 신호

### 3. 강사 데이터

- 담당 반, 시간표, 대타/휴강 이력
- 수업 품질 피드백, 민원 연관도, 학생 유지율
- 계약 형태, 정산, 업무량, 피로 신호

### 4. 운영/재무 데이터

- 수납, 미수금, 환불, 할인, 매출 추이
- 모집 채널, 상담 전환율, 등록률, 재등록률, 이탈률
- 강의실/차량/시설 가동률
- 세무 증빙, 급여/정산, 비용 구조

### 5. 케이스 데이터

- 민원 발생 시점, 주제, 심각도, 담당자, 처리 결과
- 환불/예외 승인 기록과 근거
- 재등록 방어 시도와 결과
- 문제 해결 소요 시간과 재발 여부

## 왜 중요한가

- 이 데이터가 쌓이면 `누가 위험한가`가 아니라 `어떤 운영 패턴이 문제를 만든다`를 볼 수 있다.
- 단일 LLM 응답보다 강한 차별점은 `누적 맥락`과 `예측 가능한 운영 판단`이다.
- `k-skill`은 이 데이터 위에서 domain-specific workflow로 작동할 때 가치가 커진다.

## 자산화 우선순위 제안

1. 민원/상담 로그 표준화
2. 재등록/이탈 이벤트 정의
3. 학생-학부모-강사-반 연결 그래프
4. 수납/환불/할인 승인 로그
5. 법규/점검/증빙 타임라인

## Hypotheses

- 가장 먼저 자산화할 데이터는 `성적`보다 `민원/상담/재등록/예외 승인`일 수 있다.
- 운영자가 체감하는 AI 품질은 모델 지능보다 `누적 맥락의 정확도`에 좌우된다.
- 장기적으로는 학원마다 다른 운영 know-how를 `skill + data` 형태로 패키징할 수 있다.

## Open Questions

- 최소 입력으로 가장 큰 가치를 만드는 데이터는 무엇인가?
- 어떤 데이터는 자동 수집되고, 어떤 데이터는 반드시 사람이 확인해야 하는가?
- 개인정보·민감정보·아동 데이터 처리 기준을 제품 설계에서 어디까지 선반영해야 하는가?
