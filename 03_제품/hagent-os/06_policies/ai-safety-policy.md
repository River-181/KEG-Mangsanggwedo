# Policy: AI 안전 정책

> 에이전트 자율 실행 한계, 제로휴먼 레벨, 오류 대응, 예산 관리, 권한 범위

---

## 핵심 원칙

**1. 사람이 항상 최종 결정권**
- AI는 제안만 한다. 실행은 사람이 한다 (금전·법적·인사 관련)
- 예외: Level 0 사전 승인 항목만 자동 실행

**2. 절대 자동 실행 금지 항목**
- 환불 실행
- 강사 해고 또는 계약 종료
- 법적 문서 발송 (공지·계약·약관·대출 관련)
- 학부모/강사 신원 정보 외부 공개

**3. 명확한 레벨 정책**
- Level 0: 완전 자동 (0% 인간 개입) — 사전 템플릿 승인, 조건 충족 시 자동 실행
- Level 1: 초안 + 원클릭 승인 (10% 인간 개입) — AI 초안 → 원장 승인 1회 탭
- Level 2: 초안 + 편집 후 발송 (30% 인간 개입) — AI 초안 → 원장 수정 후 발송
- Level 3: 분석 + 사람 결정 (50% 인간 개입) — AI 분석 제공, 최종 결정은 원장
- Level 4: 정보 제공만 (90% 인간 개입) — AI가 정보만 제시, 거의 모든 판단은 원장

---

## 에이전트 자율 실행 한계

### 절대 자동 실행 금지 (Level 2 이상)

**1. 환불 실행**
```
❌ 금지:
  - Complaint Agent가 "환불 처리하겠습니다" 실행
  - Finance Agent가 환불금 이체 자동 실행

✅ 필수 단계:
  1. Complaint Agent: "환불 가능 여부 판단 + 금액 계산"
  2. 대시보드: 원장 검토 및 승인
  3. 원장 클릭: "환불 승인" 버튼
  4. Finance Agent: 이체 실행 + 기록
```

**규칙:**
```
- 금액 <= 100만원: Level 1 (원장만)
- 금액 > 100만원: Level 2 (원장 + 실장 확인)
- 정책 외 환불: Level 2 (원장 + 변호사)
  예) 신뢰 잃음으로 환불 요청, 강사 퇴사로 인한 환불
```

**2. 강사 변경/해고**
```
❌ 자동 실행 절대 금지:
  - Staff Agent가 "강사 변경 실행"
  - HR 시스템에 해고 처리

✅ 프로세스:
  1. Staff Agent: "성과 평가 + 변경 필요성 분석"
  2. 대시보드: 원장이 제안 검토
  3. 원장: "강사 변경 승인"
  4. 강사/담당자에게만 알림 (학생 알림은 별도)
  5. Scheduler: 강의실 배치 변경
  6. 기록: 변경 사유 + 날짜 저장
```

**권한:**
```
- 강사 변경: 원장 또는 실장
- 강사 해고: 원장 + HR (법적 자문 권장)
- 상여금 삭감: 원장 + 실장 합의
```

**3. 법적 문서 발송**
```
❌ AI가 자동으로 보낼 수 없는 것:
  - 계약서 (신규 강사, 학생 모집)
  - 약관 변경 공지 (법적 책임)
  - 환불 거절 공식 통보
  - 고용 종료 통보 (강사)
  - 이용 정지 공지 (학생)
  - 법적 소장 관련 문서

✅ 가능한 준비 작업:
  - 문서 템플릿 작성 (검수 필요)
  - 필요 데이터 수집 및 정리
  - 변호사 검수 완료 확인
  - 대시보드에 "승인 대기" 상태로 표시

⚖️ 검수 프로세스:
  1. AI가 드래프트 작성
  2. 대시보드: 원장이 기안 검토
  3. 원장이 필요하면 변호사에게 검수 요청
  4. 변호사 수정본 적용
  5. 원장 최종 승인
  6. 발송 실행 (AI 발송 금지, 사람이 클릭)
```

## Level 0 자동 실행 허용 범위

### 명확한 기준

**허용되는 Level 0 항목:**
```
1. 연체 알림 발송 (템플릿 고정 + 사전 승인)
   - SMS: "수강료 미납 알림"
   - 카톡: "수강료 결제 안내"
   → 원장이 메시지 템플릿을 미리 승인하면 자동 발송

2. 이탈 징후 감지 (알림만, 조치 아님)
   - 대시보드에 경고 표시: "⚠️ 김민준 이탈 위험"
   → 원장은 "개입 시작" 클릭해야 실제 연락 시작

3. 감사 로그 기록 (모든 AI 작업 자동 기록)
   - 타임스탬프, 에이전트, 액션, 결과 자동 저장

4. 스케줄 확인 (읽기만, 쓰기 금지)
   - 오늘 일정 조회
   - 예외 사항 감지 (강사 부재, 강의실 문제)
   → 원장에게 알림만 하고 수정은 원장이

5. 통계 생성 (읽기 기반)
   - 월간 수강생 추이
   - 강사별 만족도
   - 수강료 추이
```

### Level 0 승인 프로세스 (최초 1회만)

```
초기 설정:

원장이 대시보드에서:
[대시보드] → [AI 자동화 설정]

체크박스:
  ☐ 연체 알림 SMS 자동 발송
    메시지: "안녕하세요. 월 수강료가 미납되었습니다. 확인 바랍니다."
    [미리보기]
    [승인] → 저장

  ☐ 카톡 연체 안내 자동 발송
    메시지: "[알림] 수강료 결제 안내"
    [미리보기]
    [승인] → 저장

→ 이후 시스템이 조건 충족 시 자동 실행
→ 감사 로그에 "Level 0 자동 실행" 기록
```

## AI 오류 대응

### 1. Hallucination (환각)

**현상:**
```
Complaint Agent가 실제와 다른 정보를 생성:
  - "강사 박수진이 퇴사했습니다" (실제: 여전히 재직)
  - "학생 김민준은 3개월 수강 없음" (실제: 2개월)
  - "정책상 환불 불가" (실제: 정책서에 그런 규정 없음)
```

**감지 및 대응:**
```
1. 학부모 또는 원장이 이의 제기:
   "이건 틀렸어요. 박수진 강사는 아직 일합니다."

2. 즉시 프로세스:
   - 대시보드: 오류 플래그 표시
   - Complaint Agent 중지 (해당 스레드)
   - 원장에게 알림: "[오류 보고] 메시지 검토 필요"

3. 검증 및 수정:
   - 원장이 실제 데이터 확인
   - 정정 메시지 원장이 직접 작성
   - 기존 메시지 회수 (가능하면)
   - 정정본 발송

4. 기록:
   - 감사 로그: 오류 원인 기록
   - AI 모델 피드백: hallucination 학습용

예시 기록:
{
  timestamp: "2026-04-09T10:23:45Z",
  error_type: "hallucination",
  agent: "Complaint",
  claim: "강사 박수진 퇴사",
  fact: "박수진 여전히 재직 (2026-01-15 계약 갱신)",
  impact: "학부모 1명 수신",
  correction_sent: "YES",
  resolved_by: "Academy-Head",
  root_cause: "직원 데이터베이스 동기화 지연"
}
```

### 2. 잘못된 법령 해석

**현상:**
```
Compliance Agent가 법령을 잘못 해석:
  - "소방 검사는 매달 해야 한다" (실제: 분기별)
  - "환불 정책상 7일 이내만 가능" (실제: 학원 정책은 14일)
```

**대응:**
```
1. 감지:
   - korean-law-mcp 라이센스 검증 (데이터 날짜)
   - 법령 변경 이력 확인
   - 학원 내부 정책과 비교

2. 수정:
   - 법령 데이터 최신화 (korean-law-mcp 업데이트)
   - Compliance Agent 지시문 갱신
   - 과거 오류 권고사항 일괄 검수

3. 학부모 피해 보정:
   - "AI의 잘못된 안내로 불편을 드렸습니다"
   - 정정 안내 발송
```

## 예산 초과 시 자동 일시정지

### 비용 모니터링

**시나리오 1: Claude API 과다 호출**
```
설정:
  월 예산 한도: 100만원 (API 비용)
  경고 임계값: 80% (80만원)

실시간 모니터링:
  [대시보드] → [AI 비용 현황]
  - 현재 월 누적: 72만원 (72%)
  - 오늘 예상: +18만원
  - 예상 월말 합계: 90만원 (90%)

경고 트리거:
  1. 80만원 도달 → 원장에게 알림
  2. 원장 승인 필요: "추가 비용 인정?"
  3. 100만원 도달 → 자동 일시정지
     (긴급 작업만 수동으로 재개 가능)
```

**시나리오 2: 에이전트 폭주**
```
예: Retention Agent가 무한 루프
  (1시간에 API 호출 10,000회 = 예정의 10배)

자동 감지:
  1. API 호출 속도 이상 (초과 감지)
  2. 즉시 Retention Agent 중지
  3. 원장에게 긴급 알림
  4. 에러 로그 생성

원장 액션:
  [재개 승인] 또는 [종료]
  (원인 분석 후 재설정 필요)
```

## 에이전트 권한 범위

### 역할별 권한 매트릭스

| 에이전트 | 읽기 | 쓰기 | 실행 |
|---------|------|------|------|
| **Complaint** | ✅ 민원 | ✅ 분류 + 초안 | ❌ 발송 금지 |
| **Retention** | ✅ 학생 로그 | ✅ 위험 플래그 | ❌ 연락 금지 |
| **Compliance** | ✅ 법령 + 공문 | ✅ 기한 기록 | ❌ 공식 통보 금지 |
| **Finance** | ✅ 결제 기록 | ✅ 환불 계산 | ❌ 이체 금지 |
| **Staff** | ✅ 강사 평가 | ✅ 분석 | ❌ 인사 결정 금지 |
| **Scheduler** | ✅ 일정 | ✅ 예약 등록 | ⚠️ 정규 일정만 (예외는 원장 승인) |
| **Orchestrator** | ✅ 모든 데이터 | ✅ 종합 | ❌ 실행 금지 (원장 지시만 받음) |

## 사람이 항상 Override 가능

### 원장의 거부권

**어떤 AI 제안도 거절 가능:**
```
원장: "AI가 제안한 강사 변경을 거절합니다."

→ 시스템:
   1. 거절 사유 입력 (선택)
   2. 기록: "2026-04-09 10:23, 강사 변경 제안 거절, 사유: 추가 관찰 필요"
   3. 해당 에이전트는 다음 제안 때 거절 이유 참고
   4. 수동으로 다시 제안해도 됨
```

**AI 실행 중단 (긴급 정지):**
```
원장이 대시보드에서:
[긴급 중단 버튼] → 모든 에이전트 즉시 정지

예시:
  - API 비용이 급상승 중
  - 이상한 메시지가 발송되고 있음
  - 강사 관계에 문제 발생
```

**과거 명령 무효화:**
```
원장: "어제 환불 승인한 것 취소하고 싶습니다."

→ 시스템:
   1. 이미 이체됨 → 회수 프로세스 시작
   2. 이체 전 → 취소 처리
   3. 기록: "취소 사유 + 책임자 + 역거래"
```

## 성공 지표

| 지표 | 목표 |
|------|------|
| **오류 발생** | < 0.1% (AI 제안 중) |
| **오류 감지율** | 100% (학부모 클레임 0) |
| **오류 수정 시간** | < 2시간 |
| **원장 거부율** | 5~10% (정상 범위) |
| **예산 초과** | 0건 / 월 |
| **권한 침범** | 0건 / 연 |
