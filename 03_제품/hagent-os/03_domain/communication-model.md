# Communication Model (커뮤니케이션 도메인)

> 학원 내외부 소통 구조, 민원 처리, 성적 리포트, 조직 역할 도메인 지식.
> 제품 스펙이 아닌 도메인 레퍼런스. HagentOS 에이전트 설계의 도메인 기반.

---

## 커뮤니케이션 채널 현황

한국 학원의 학부모 소통은 카카오톡 생태계에 ==사실상 표준화==되어 있다.

| 채널 | 특성 | 주요 문제 |
|------|------|-----------|
| 카카오톡 개인 DM | 교사·원장 개인 전화번호 노출, 실시간 응답 압박 | 24시간 울림, 업무·사생활 경계 없음 |
| 카카오 단톡방 | 반 단위 공지, 학부모 그룹 소통 | 중요 메시지 묻힘, 확인 불가 |
| 카카오 채널(알림톡) | 출결·수납 자동 알림, 공식 발신 | 단방향, 개별 응대 불가 |
| 전화 | 긴급 민원, 상담 약속 | 기록 없음, 야간 전화 피로 |
| SMS | 대체 알림 (알림톡 실패 시) | 단방향, 맥락 없음 |
| 앱 내 메시지 (클래스팅 등) | 공지·알림장 특화 | 도입 학원 제한적, 이중 입력 발생 |

**핵심 문제**: "결국 카카오톡과 엑셀로 돌아간다" — 전용 앱을 쓰더라도 실제 소통은 카톡으로 흐른다. 개인 채널과 공식 채널이 분리되지 않아 맥락이 흩어진다.

## 학부모 소통 유형

학원과 학부모 간 소통은 크게 7가지 유형으로 분류된다.

### 1. 일상 공지
- 수업 변경, 휴강, 행사, 시험 일정
- 현재 방식: 단톡방 공지 또는 알림톡 자동 발송
- 반복 발생, 자동화 가능 구간

### 2. 출결 알림
- 등원·하원 확인, 결석 사전 통보, 지각 알림
- 현재 방식: 알림톡 자동화 비교적 잘 됨 (클래스업, 온하이 등)
- ==자동화 커버율 가장 높은 영역==

### 3. 숙제·수업 리포트
- 당일 수업 내용, 숙제 안내, 수행 피드백
- 현재 방식: 교사 개인 카톡, 알림장 앱
- 반복 입력이 많고 자동화 수요 높음

### 4. 성적 리포트
- 시험 후 점수·분석·피드백 전송
- 현재 방식: 엑셀·한글 성적표 수동 작성 후 개별 카톡 — 5시간/주(시험 기간 기준)
- 온길 OMS 사례: 점수 입력만 하면 자동 리포트 생성 → 1시간/주로 단축

### 5. 상담 요청
- 학부모 주도 문의, 원장·강사 주도 면담 제안
- 현재 방식: 카톡 DM → 전화 → 방문
- 상담 기록이 시스템에 남지 않는 경우 다수

### 6. 수납 관련
- 청구서 발송, 미납 안내, 환불 협의
- 현재 방식: 알림톡 자동 발송 도입 학원 많음. 미납 독촉은 전화·카톡
- 수납·성적·상담이 전체 행정 시간의 60% 차지 (온길 OMS 데이터)

### 7. 민원
- 성적 불만, 강사 교체 요구, 환불, 스케줄 조정 등
- ==현재 방식: 원장 직접 처리, 시스템 기록 없음==
- 가장 비정형이고 자동화 공백이 큰 영역

## 민원 분류 체계

### 민원 유형

| 분류 | 예시 | 심각도 | 대응 주체 |
|------|------|--------|-----------|
| 성적 불만 | 점수 하락, 강사 수업 품질 | 중~고 | 강사 + 원장 |
| 행동 지적 | 학생 태도, 강사 언행 | 중 | 원장 |
| 환불 요청 | 중도 퇴원, 과납 환급 | 고 | 원장 |
| 스케줄 조정 | 시간대 변경, 반 이동 | 저~중 | 실장 + 원장 |
| 강사 교체 요구 | 궁합, 수업 방식 불만 | 고 | 원장 |
| 시설·안전 불만 | 냉난방, 차량, 공간 | 중 | 원장 + 시설 |
| 기타 공지 오류 | 잘못된 정보, 일정 혼선 | 저 | 실장 |

### 민원 처리 흐름

```
접수 (카톡 DM / 전화) 
  → 분류 (성격·심각도 파악)
  → 1차 대응 (실장 또는 원장)
  → 사실 확인 (강사·시스템 데이터 조회)
  → 의사결정 (환불 여부, 강사 교체 등)
  → 학부모에 결과 통보
  → 후속 모니터링 (이탈 여부, 재발 방지)
```

### 야간·주말 민원 문제

- 학원은 저녁·주말에 운영 → 민원도 저녁·주말에 집중
- 원장 개인 카톡으로 접수 → ==즉시 응답 압박==
- 기록이 없으니 같은 민원 반복 발생해도 추적 불가
- 교사 학부모 민원 스트레스 경험 **56.9%** (OECD TALIS 2024, 세계 2위)

## 성적·학습 리포트

### 성적 리포트 제작 현재 흐름

```
시험 종료
  → 교사 채점
  → 점수 엑셀 입력
  → 성적표 한글/워드 작성 (학생별)
  → 개별 카톡 전송 or 출력 배부
```

소요 시간: 시험 기간 기준 5시간/주. 클래스당 20~30명이면 가중됨.

### 학습 리포트 데이터 구성

- 점수 (원점수·표준점수·백분위)
- 오답 분석 (단원·유형별)
- 전회 대비 변화
- 학부모 메시지 (강사 코멘트)

### 수능·모의고사 일정 연동

- 한국교육과정평가원(KICE) 공지 → PDF/HWP 형식
- 학원이 별도로 파싱 → 내부 캘린더·공지로 재작성
- 6월·9월 모의평가, 수능, 각종 학력 검사 기준 일정이 학원 운영 리듬을 결정

## 내부 소통 구조

### 조직 계층별 소통

```
원장
  ↕ (경영 판단·승인 요청)
실장 (또는 수석 강사)
  ↕ (일정·민원·수납 조율)
강사
  ↕ (학생 상태·수업 피드백)
학생 / 학부모
```

### 회의·보고 체계

- 중소형 학원(2~5인): 비공식 구두 보고, 카카오톡 채팅방
- 중형 학원(5~20인): 주간 회의, 수업 리포트 서면 제출
- 대형 프랜차이즈: 지점 → 본사 주간 실적 보고, 표준 양식

### 강사 → 원장 보고 루틴

- 학생 이상 징후 (결석 반복, 성적 급락)
- 학부모 불만 접수 사실
- 수업 변경 필요 사항 (보강, 자료 요청)

현재 방식: 카카오톡 채팅방 or 구두. ==기록이 남지 않아 원장이 민원 전까지 파악 못하는 경우 多==.

## 조직 역할별 소통 패턴

| 역할 | 주요 소통 대상 | 소통 빈도 | 비고 |
|------|---------------|-----------|------|
| 원장 | 학부모 (민원), 강사 (관리), 실장 (운영), 세무/노무 | 전방위, 고빈도 | 의사결정 병목 |
| 실장 | 학부모 (접수·일정), 강사 (일정 조율), 원장 (에스컬레이션) | 중간 허브 | 없는 학원도 많음 |
| 강사 | 학생 (수업), 학부모 (숙제·리포트 직접), 원장 (보고) | 수업 후 집중 | 개인 카톡 노출 위험 |
| 행정/조교 | 학부모 (수납·공지), 강사 (자료 지원) | 정형 업무 중심 | 소형 학원은 원장이 겸임 |

**원장의 소통 병목**: 민원 최종 결재, 환불 승인, 강사 평가, 지점 간 조율 — 모두 원장 직접 처리 구조. 하루 행정업무 3시간 = 월 300만원 기회비용 손실 (bati.ai 분석).

## 소통의 Pain Points

### 구조적 문제

- **채널 분절**: 공지는 알림톡, 상담은 카톡, 성적은 카톡, 민원도 카톡 → 학생 맥락이 채널별로 흩어짐
- **맥락 유실**: 상담 기록이 시스템이 아닌 카톡 대화 속에만 존재 → 강사 교체·원장 부재 시 인수인계 불가
- **이중 입력**: 학원 ERP에 출결 입력 + 카톡으로 학부모에게 개별 공지 = 같은 정보를 두 번 작성
- **기록 부재**: 민원 케이스, 상담 약속, 환불 협의 내용이 카톡에만 남고 집계 불가

### 운영자 감정 노동

- 24시간 응대 압박: "밤에도 카톡이 울린다" — 수업 끝나고 폰 붙잡는 시간이 1~2시간
- 학부모 감정 관리: 민원 시 원장이 직접 사과·설득 → ==필코노미(학부모 감정 관리가 학원 생존의 핵심)==
- 강사 스트레스: 개인 번호 노출 → 퇴근 후 업무 연속 → 강사 이직률 높음

### 솔루션 도입 후에도 남는 문제

- 알림톡 자동화를 써도 비정형 민원·상담은 카톡으로 빠짐
- ERP를 써도 "메뉴가 많고 현장 예외 처리가 느리고 복잡하다"는 불만 반복
- 결과적으로 ERP + 카카오톡 + 엑셀 삼중 구조가 유지됨

## 데이터 자산으로서의 소통 기록

소통 이력은 단순 로그가 아니라 ==학원 운영 인텔리전스의 원천==이다.

### 축적되면 가능해지는 것

| 데이터 | 축적 시 가능한 판단 |
|--------|-------------------|
| 민원 이력 (시점·유형·심각도·결과) | 반복 민원 원인 진단, 강사 품질 지표, 서비스 품질 지수 |
| 학부모 성향 (반응 속도, 민감 포인트, 약속 이력) | 이탈 위험 예측, 맞춤 대응 전략 |
| 상담 기록 (문의 → 등록 → 재등록) | 전환율 분석, 재등록 방어 타이밍 파악 |
| 강사–민원 연관도 | 강사별 고객 만족도 산출, 인사 의사결정 보조 |
| 케이스 처리 시간 | 운영 효율성 측정, 병목 식별 |

### 현재 자산화가 안 되는 이유

- 카톡 대화는 구조화 불가 (텍스트 덩어리)
- ERP 상담 메모는 자유 입력 → 검색·분석 어려움
- 민원 결과(환불 여부, 강사 교체 등)가 시스템에 기록되지 않음
- 채널별 기록이 분산 → 단일 학부모 뷰 생성 불가

> 가장 먼저 자산화할 데이터: 성적보다 **민원/상담/재등록/예외 승인** 로그 (data-assets.md 분석)

## HagentOS 매핑 힌트

이 도메인 지식이 에이전트 설계에 연결되는 방식. (제품 스펙 아님 — 도메인 인사이트 메모)

| 도메인 패턴 | 연관 에이전트 |
|------------|--------------|
| 민원 접수·분류·1차 응대 자동화 | `Complaint Agent` |
| 이탈 징후 감지 + 학부모 재등록 유도 | `Retention Agent` |
| 성적 리포트 자동 생성·전송 | `Analytics Agent` |
| 출결·공지·숙제 알림 자동화 | `Notification Agent` |
| 신규 상담 문의 접수·분류·예약 연결 | `Intake Agent` |
| 내부 보고 (강사 → 원장) 구조화 | `Orchestrator` |

**에이전트 설계 원칙 시사점**:
- 모든 소통 에이전트는 학부모·학생·강사 맥락을 단일 뷰로 통합해야 함
- 민원 에이전트는 야간·주말 접수 → 분류 → 원장 승인 요청 플로우가 핵심
- 리포트 에이전트는 카카오 알림톡 발송 연동이 필수 (84% 채널 점유율)
- 데이터 축적 없이 에이전트는 기억이 없다 — ==케이스 로그가 에이전트의 장기 기억==
