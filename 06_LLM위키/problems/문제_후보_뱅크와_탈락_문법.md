# 문제 후보 뱅크와 탈락 문법

## Summary First

문제 은행의 핵심은 후보를 많이 모으는 것이 아니라, 무엇을 남기고 무엇을 버렸는지의 기준을 명확히 남기는 것이다. 이 문서는 현재 살아 있는 후보와 탈락 규칙을 함께 요약한다.

## Live Candidates

### P-001 학원 원장 운영 보조 Swarm

- 사용자: 사교육 운영자
- 핵심 pain: 민원, 강사 관리, 이탈 예측, 상담/예외 처리
- 포지셔닝: 현재 가장 강한 코어 후보

### P-002 담임교사 반 운영 Swarm

- 사용자: 공교육 담임교사
- 핵심 pain: 민원, 행정, 생기부, 수업 계획
- 포지셔닝: 강한 대안이지만 현재 중심축은 아님

### P-003 학원 신규 상담 자동화

- 사용자: 학원 원장/상담 담당
- 핵심 pain: 문의 응답 지연, 상담 예약 누락, 반 배정 수작업
- 포지셔닝: P-001의 확장 장면으로 넣기 좋다

### P-004 학부모 민원 지능형 처리

- 사용자: 공교육/사교육 cross-cutting
- 핵심 pain: 비정형 민원, 반복 재응대, 감정 노동
- 포지셔닝: 독립 제품보다 코어 기능 축

## White Space

- 역할 기반 AI 에이전트 오케스트레이션
- 신규 상담 -> 등록 -> 이탈 예측의 연결 workflow
- 비정형 민원의 맥락 파악 + 응답 초안 + 에스컬레이션
- 강사 성과/피로도 자동 분석

## Rejection Rules

- 이미 강한 incumbent가 있고 7일 내 차별화가 어려우면 버린다.
- IoT, 실제 현장 연동, 고도 규제 의존으로 MVP 리스크가 크면 버린다.
- `출결/수납 자동화`처럼 레드오션이면 버린다.

## Rejected Now

- 생기부 단독
- 차량 관리 단독
- 출결·수납 자동화 단독

## Strategic Reading

- 살아 있는 후보는 모두 `운영 루틴` 문제다.
- 단순 챗봇이 아니라 역할 기반 분담 처리 구조가 공통 차별점이다.
- 문제 후보를 볼 때 "독립 제품"인지 "코어 기능"인지를 구분해야 한다.
