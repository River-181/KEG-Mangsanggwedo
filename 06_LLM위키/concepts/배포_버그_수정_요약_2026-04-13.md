# 배포 버그 수정 요약 2026-04-13

## Summary First

2026-04-13의 배포 안정화는 “기능 추가”보다 “라이브와 로컬의 기대치를 일치시키는 수정”이 핵심이었다. 특히 `server typecheck blocker`, `health endpoint mismatch`, `org-aware adapter status`, `Vite port/proxy coupling`, `SSE reconnect`가 주된 버그 축이었다.

## 핵심 수정 묶음

| 버그 축 | 원인 | 수정 | 검증 상태 |
|---------|------|------|-----------|
| server typecheck blocker | `AgentType` union에 없는 `"ceo"` 비교 | `server/src/routes/organizations.ts`에서 비교 제거 | 통과 |
| health check 혼선 | `/`와 `/api/health` 기대가 달랐음 | root `/`를 health JSON alias로 정리 | 통과 |
| Settings의 adapter 오판 | 전역 env 기준으로만 상태를 보여줌 | `orgId`를 받는 `/api/adapters`와 org-aware `SettingsPage`로 수정 | 통과 |
| Vite / proxy 포트 꼬임 | frontend port와 backend target이 함께 엮였음 | `UI_PORT`/`VITE_PORT`, `API_PORT`, `strictPort`로 분리 | 통과 |
| live restart / dev command 혼선 | 실행 명령과 기본 포트 기대가 흔들림 | root `pnpm dev` 정리 + server 기본 `3200` 기준 고정 | 통과 |
| SSE stale reconnect | offline/visibility 복귀 후 stale 상태 잔존 | exponential backoff, `online`, `visibilitychange`, query invalidate 보강 | partial |

## 실제로 고쳐진 운영 문제

### 1. `server typecheck`가 막히던 상태 해소

- 증상: `organizations.ts`에서 `agentType === "ceo"` 비교로 `TS2367`
- 해석: type 정의와 실제 seed/slug 규칙이 어긋난 상태
- 결과: `server typecheck` 재통과

### 2. health endpoint 기대치 통일

- 증상: 어떤 문서는 `/`, 어떤 체크는 `/api/health`를 기대
- 수정: 둘 다 health JSON을 반환하도록 정리
- 의미: smoke check와 운영 확인이 덜 헷갈리게 됨

### 3. 기관별 Telegram readiness 표면 보정

- 증상: Settings에서는 조직에 bot token이 있어도 degraded처럼 보일 수 있었음
- 수정: `GET /api/adapters?orgId=...`가 조직 설정을 반영
- 결과: live 기준 `telegram-outbound.connected = true`가 확인됨

### 4. Vite 포트와 backend proxy target 분리

- 증상: dev server 포트와 API proxy가 같은 환경 변수에 묶여 재기동 시 혼선
- 수정: frontend / backend 포트 경계를 분리
- 결과: `5174` UI와 `3200` API 기준이 다시 명확해짐

### 5. SSE reconnect 보강

- 증상: 탭 복귀나 일시적 네트워크 오류 뒤 stale 상태가 오래 남을 수 있었음
- 수정: reconnect backoff, `online`, `visibilitychange`, query invalidate 추가
- 주의: event stream 자체는 확인됐지만 브라우저 수준 end-to-end refresh는 아직 부분 검증이다

## 이 라운드에서 함께 정리된 것

- `Settings`의 integration status 출처 표기 보강
- `Dashboard`와 `Inbox`의 보조 패널 밀도 조정
- live `3200` 재기동 후 `/`, `/api/health`, `/api/adapters?orgId=...` 재확인

## 검증에 사용된 대표 체크

- `npm exec pnpm -- --filter @hagent/server typecheck`
- `corepack pnpm --filter @hagent/ui exec tsc --noEmit ...`
- `corepack pnpm --filter @hagent/ui exec vite build`
- `curl http://127.0.0.1:3200/`
- `curl http://127.0.0.1:3200/api/health`
- `curl 'http://127.0.0.1:3200/api/adapters?orgId=be70ebc8-3b55-4ff3-827a-264f06c4d2ee'`

## 남은 리스크

- SSE reconnect가 실제 브라우저 데이터 refresh까지 이어지는지 완전 증적은 부족하다.
- Telegram/Kakao 실제 외부 outbound receipt 증적은 여전히 부족하다.
- 몇몇 external env는 여전히 credential 의존 상태다.
