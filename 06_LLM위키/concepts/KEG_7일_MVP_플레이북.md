# KEG 7일 MVP 플레이북

## Core Thesis

이 대회는 AI를 많이 쓴 팀보다 `문제정의 -> 구현 -> 배포 -> 증빙 -> 설명`을 짧은 시간 안에 끝낸 팀이 유리하다. 따라서 전략의 중심은 넓은 기능이 아니라 빠른 결정과 demo-first 범위 관리다.

## Operating Principles

- 문제는 아이디어가 아니라 `문제 지도`로 준비한다.
- 주제 공개 후 6~12시간 안에 핵심 사용자, 핵심 순간, 데모 장면, 버릴 기능을 확정한다.
- 고정된 시간, 가변 범위를 전제로 must/should/could/not-now를 공격적으로 자른다.
- Day 3 이후 새 기능 추가보다 안정화와 증빙을 우선한다.

## Decision Questions

- 이 기능이 없어도 데모가 되는가
- 이 기능이 없어도 문제 정의가 살아 있는가
- 이 기능은 AI가 아니어도 되는가
- 이 기능은 배포 실패를 부르는가
- 이 기능은 README와 AI 활용 리포트에 설명 가능한가

## Method Stack

- YC / Make something people want
- Lean Startup / Build-Measure-Learn
- Design Sprint / 빠른 의사결정과 프로토타입
- Shape Up / fixed time, variable scope

## What This Changes For Us

- 제품은 `운영 루틴 하나`를 날카롭게 해결해야 한다.
- 문제 후보는 주제 공개 전에 준비하고, 공개 후에는 selection과 cutting이 중심이 된다.
- 데모 대본을 구현 전에 먼저 쓰는 편이 맞다.
