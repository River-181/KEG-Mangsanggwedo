# 학원 운영 루프 지도

## Summary First

학원 운영의 본체는 정규 데이터 처리보다 `민원`, `보강`, `재등록`, `강사 스케줄`, `환불`, `세무`, `법규`, `안전` 같은 예외와 책임 업무다. 따라서 제품도 정규 ERP보다 `case + approval + follow-up` 중심으로 설계해야 한다.

## Daily Loop

- 신규 문의 응대와 상담 일정 조율
- 결석/지각 확인, 보강 조정, 시간표 예외 처리
- 학부모 메시지/전화 응답
- 강사 공결/대타/강의실 배정 조정
- 출결/미납/현장 문의 처리
- 학생 상태 공유: 성적, 태도, 이탈 징후
- 차량형 학원은 귀가/보호자 커뮤니케이션 추가

## Weekly Loop

- 주간 상담/민원 정리와 재응대
- 등록률, 재등록률, 결석 추세 점검
- 강사 스케줄과 업무량 재조정
- 반편성, 보강, 휴강, 대체수업 확정
- 교재/평가 준비와 결과 공유
- 학부모 공지, 행사/시험 메시지 발송

## Monthly Loop

- 수납 마감, 미수금 추적, 환불/할인/예외 승인
- 급여/프리랜서 정산, 세무 자료 정리
- 사업장현황신고, 현금영수증, 면세 매출 증빙 준비
- 신규 모집 성과와 재등록/이탈 분석
- 강사 계약, 채용/퇴사, 근로자성 리스크 점검
- 시설, 소방, 통학버스, 보험, 법정교육 일정 점검

## Exception-driven Work

- 성적 불만, 강사 교체 요구, 환불 분쟁
- 심야 감정 민원과 약속 이행 추적
- 차량 사고/지연, 시설 고장, 안전 이슈
- 교육청 점검, 세무/노무 문의 대응

## Product Reading

- 구조화된 `출결/수납/공지`보다 비정형 운영 case가 더 중요하다.
- 운영자가 원하는 것은 업무 대행 자체보다 `맥락 복원 + 승인 전 정리 + 사후 기록`일 가능성이 높다.
- 공통 코어는 `학생/학부모/강사/케이스/결제/일정/승인/로그`다.

## Initial Loop Hypothesis

1. 문의/이슈 접수
2. 학생/학부모/강사 맥락 조회
3. 초안/안내/대응안 생성
4. 사람 승인
5. 발송/실행
6. 케이스 로그와 후속 일정 저장
