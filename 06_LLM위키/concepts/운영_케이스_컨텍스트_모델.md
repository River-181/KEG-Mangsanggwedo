# 운영 케이스 컨텍스트 모델

## Summary First

운영 시스템의 핵심 context는 대화문맥이 아니라 업무 상태다. 민원, 상담, 보강, 환불, 재등록 위험은 메시지 조각이 아니라 사람, 시간, 규칙, 이력, 현재 상태가 연결된 `case`로 다뤄져야 한다.

## Model Shift

- chat context -> operational state
- message thread -> case history
- single response -> tracked run
- vague memory -> audit log

## First-Class Objects

- 조직 / 지점 / 반
- 학생 / 학부모 / 강사
- 운영 케이스
  - 민원
  - 상담
  - 보강
  - 환불
  - 결석 대응
- 정책 / 약관 / 컴플라이언스 기준
- activity / audit log

## Why It Matters

- 민원은 감정 이력과 약속 이력이 붙어야 한다.
- 강사 일정 조정은 workload와 규칙을 읽어야 한다.
- 재등록 위험은 출결, 성과, 상담 기록, 불만 신호가 연결돼야 한다.
- 따라서 답변 품질보다 상태 모델이 먼저다.

## Recommended Storage View

- master data:
  - 학생
  - 학부모
  - 강사
  - 반
  - 지점
- operational case:
  - 민원
  - 상담
  - 보강
  - 환불
  - 결석 대응
- policy/compliance:
  - 약관
  - 규정
  - 법무 체크 기준
- audit/activity:
  - 누가 언제 무엇을 바꿨는지

## Guardrails

- company schema를 그대로 복제하지 않는다.
- chat history만으로 운영 시스템처럼 보이게 만들지 않는다.
- 모든 학원 유형을 같은 entity model로 억지로 맞추지 않는다.

## Open Design Question

- `민원`을 conversation, ticket, case 중 어느 개념으로 고정할 것인가
- MVP에서 필수 data object는 무엇인가
- 데이터 자산화 스토리를 어디서 시작할 것인가
