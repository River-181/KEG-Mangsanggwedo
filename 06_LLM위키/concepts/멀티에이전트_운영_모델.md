# 멀티에이전트 운영 모델

## Summary First

이 프로젝트의 실제 운영 모델은 “한 AI가 모든 걸 하는 방식”이 아니다. 핵심은 `Claude + GPT + Codex + Perplexity`를 역할별로 병렬 배치하고, `PROGRESS.md`를 단일 진실 소스로 두며, 모든 세션을 `intake-first`로 기록하는 협업 운영체계다.

## canonical 4-track

> 실제 전체 기록에는 Gemini, Grok, NotebookLM도 등장하지만, 현재 운영 정본을 4축으로 압축하면 아래가 핵심이다.

| 축 | 대표 도구/모델 | 현재 역할 | 대표 산출물 |
|----|----------------|-----------|-------------|
| Builder | Claude Code / Opus 4.6 / Sonnet 4.6 | 구현, 리팩터, 디버깅 | app code, handoff, replay 가능한 수정 |
| PM/Product | GPT-5.4 | 스코프 결정, 구조화, 문서 합성 | 전략 문서, 플레이북, 제품 framing |
| Reviewer/Evidence | Codex / GPT-5 / GPT-5.4 | 코드 감사, 구조 리뷰, 증빙 운영 | review findings, LLM wiki, intake CSV 운영 |
| Research | Perplexity | 출처 기반 외부 리서치 | 기관/경쟁/시장/생태계 소스 |

## 이 모델이 돌아가는 이유

### 1. `PROGRESS.md`가 단일 진실 소스다

[[.agent/system/ops/PROGRESS]]는 다음 역할을 동시에 맡는다.

- 모든 에이전트가 시작 전에 읽는 파일
- 현재 단계와 블로커를 고정하는 파일
- 병렬 AI 사이 중복 작업과 충돌을 줄이는 조율 표

즉 병렬성은 자유롭게 열어두되, 우선순위와 완료 기준은 한 파일에 수렴시킨다.

### 2. 기억을 두 층으로 나눈다

- [[.agent/system/memory/long-term-memory]]: 제품 정체성, 기술 스택, canonical org, 운영 규칙
- [[.agent/system/memory/daily-memory]]: 최근 세션 handoff, 오늘의 블로커, dirty worktree 주의점

그리고 그 사이의 재사용 가능한 합성층이 [[index]]를 시작점으로 하는 `06_LLM위키/`다.

### 3. intake-first 로깅을 강제한다

기록은 회고용이 아니라 재현성용이다.

1. 세션 종료
2. `ai-session-intake.csv`에 row append
3. `dispatch-session-intake.py` 실행
4. `master-evidence-ledger.md`와 usage 집계 재생성

이 구조 덕분에 “누가 어떤 모델로 무엇을 만들었는가”가 나중에도 복원된다.

## 병렬 원칙

### 병렬로 여는 것

- 구현과 리뷰
- 전략과 리서치
- 문서 합성과 증빙 정리

### 직렬로 잠그는 것

- 최종 제품 방향
- commit 단위 변경
- 제출물 정본

한 줄로 요약하면 `parallel by role, serial by source of truth`다.

## 현재 프로젝트에 맞는 해석

- Claude는 주 구현자다.
- GPT-5.4는 상위 구조화와 PM/Product 쪽이 강하다.
- Codex는 독립 감사자이자 위키/증빙 유지자 역할이 강하다.
- Perplexity는 출처가 필요한 외부 사실 확보에 최적화되어 있다.

이렇게 나누면 각 도구의 약점이 다른 도구의 강점으로 상쇄된다.

## 운영 프로토콜

1. 작업 시작 전 [[.agent/system/ops/PROGRESS]] 확인
2. 필요하면 [[.agent/system/memory/long-term-memory]]와 [[.agent/system/memory/daily-memory]] 확인
3. durable synthesis는 `06_LLM위키/`에 writeback
4. 세션 종료 시 intake CSV 기록
5. 제출물/핸드오프는 다시 `PROGRESS`와 memory에 반영

## 리스크 제어

- `PROGRESS`를 건너뛰면 중복 구현이 생긴다.
- intake를 생략하면 재현성과 AI 활용 증빙이 약해진다.
- long-term / daily / wiki 레이어를 섞어 쓰면 기억의 위계가 무너진다.
