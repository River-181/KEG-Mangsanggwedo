# 바이브코딩 공모전 도메인 키워드 미니 사전

> source type: moved raw research note
> summary page: [[도메인_지식_조사_03_요약]]

> 이 문서는 공모전 준비용으로, 각 도메인에서 자주 나올 개념을 "한 줄 정의 + 대회 맥락 메모" 수준으로 정리한 초안입니다.

---

## 1. 교육·학습 이론 도메인

### 교육(Education)
- 사람과 사회가 지식·기술·가치를 다음 세대로 전달하는 제도적·사회적 활동 전체를 의미합니다.[web:14]
- 공모전 맥락에서는 "우리가 어떤 방식의 교육 경험을 제공하려 하는지"를 설명하는 가장 바깥 프레임입니다.[web:14]

### 평생학습(Lifelong learning)
- 학교 시기를 넘어 생애 전 주기에 걸쳐 계속해서 학습이 일어나는 과정을 말합니다.[web:14]
- 성인 코딩 학습자, 직장인 리스킬링 같은 시나리오를 설계할 때 근거 개념으로 활용할 수 있습니다.[web:14]

### 형식·비형식·비공식 교육(Formal/Non‑formal/Informal)
- 형식 교육은 학교·대학처럼 공식 커리큘럼과 자격이 있는 제도권 교육, 비형식은 기관이 있지만 자격이 아닌 역량 중심 프로그램, 비공식은 일상 속 자발적 학습을 뜻합니다.[web:14]
- "우리 솔루션이 어느 영역에 위치한 학습 경험인지"를 포지셔닝할 때 유용합니다.[web:14]

### 학습(Learning)·학습자 중심(Learner‑centered)
- 학습은 경험으로 인한 비교적 지속적인 행동·지식 변화이며, 학습자 중심은 교사가 아니라 학습자의 목표·선호·속도를 중심에 두는 설계 관점입니다.[web:14]
- LLM 튜터나 맞춤 문제 추천을 설계할 때 "학습자 중심"이라는 키워드를 근거로 삼을 수 있습니다.[web:14]

### 자기주도학습(Self‑directed learning)
- 학습자가 목표 설정, 계획, 실행, 평가를 스스로 조정하며 학습을 이끄는 상태를 말합니다.[web:14]
- 코딩 연습 경로를 스스로 설계하게 하되 AI가 메타 피드백을 주는 구조를 설계할 때 중요한 이론입니다.[web:14]

### 교수(Teaching)·교수 설계(Instructional design)
- 교수는 학습이 잘 일어나도록 의도적으로 돕는 활동, 교수 설계는 그 구조와 전략을 체계적으로 설계하는 과정입니다.[web:14]
- 기능을 나열하기보다 "어떤 순서와 피드백 구조로 가르치는지"까지 설명할 때 설득력이 높아집니다.[web:14]

### 교육과정(Curriculum)·역량기반 교육(Competency‑based education)
- 교육과정은 무엇을 언제 어떻게 가르칠지에 대한 총괄 계획이고, 역량기반 교육은 시간·학점보다 학습자가 실제로 갖춘 능력과 수행으로 성취를 판단합니다.[web:14]
- 난이도 조정·스킬 트리·뱃지 설계에 역량기반 관점을 적용하면 "완료" 대신 "할 줄 아는 것"에 초점을 맞출 수 있습니다.[web:14]

### 행동주의(Behaviorism)
- 보상과 처벌 같은 외부 자극과 그에 따른 관찰 가능한 행동 변화를 중심으로 학습을 설명하는 이론입니다.[web:5]
- 정답·오답 피드백, 반복 드릴, 점수·뱃지 같은 구조는 행동주의적 설계 사례로 볼 수 있습니다.[web:5]

### 인지주의(Cognitivism)
- 인간을 정보 처리기로 보고, 기억·주의·문제해결 같은 내부 인지 과정을 조직해 줌으로써 학습을 돕는 이론입니다.[web:5]
- 단계적 설명, 예시–비예시, 개념 맵 같은 기능은 인지주의 관점의 기능 근거가 됩니다.[web:5]

### 구성주의·사회적 구성주의(Constructivism/Social constructivism)
- 학습자가 기존 지식과 경험을 바탕으로 능동적으로 의미를 구성한다는 관점이며, 사회적 구성주의는 상호작용과 협력을 통해 지식이 공동 구성된다고 봅니다.[web:5][web:14]
- 프로젝트 기반 미션, 페어 프로그래밍, 토론 게시판 등은 구성주의·사회적 구성주의에 기반한 학습 경험 사례입니다.[web:5]

### 인지부하(Cognitive load)
- 인간 작업 기억의 용량이 제한되어 있다는 전제에서, 한 번에 처리해야 할 정보량이 많아지면 학습 효율이 떨어지는 현상을 말합니다.[web:14]
- 한 화면에 정보·코드·힌트를 과하게 쌓지 않고, 단계적으로 드러내는 UI 설계 근거로 활용할 수 있습니다.[web:14]

### 형성 평가(Formative assessment)
- 수업·학습 과정 중에 진행되며 다음 학습을 조정하기 위한 진단·피드백 목적의 평가입니다.[web:20][web:23]
- 자동 힌트, 중간 체크퀴즈, 진단 테스트 등은 형성 평가를 디지털로 구현한 예시가 됩니다.[web:23]

### 총괄 평가(Summative assessment)
- 단원·과정이 끝난 시점에 최종 성취 수준을 판단하기 위한 고스테이크 평가를 말합니다.[web:20][web:29]
- 최종 프로젝트, 코스 완료 시험, 인증 테스트 등은 총괄 평가에 해당하며 공모전 심사와도 연결됩니다.[web:29]

### 러닝 아웃컴(Learning outcomes)·루브릭(Rubric)
- 러닝 아웃컴은 학습 후 학습자가 할 수 있게 되기를 기대하는 구체적 성취 진술이고, 루브릭은 그 수준을 구분하는 평가 기준표입니다.[web:23]
- 기능 설명을 "OO를 할 수 있다" 수준으로 쓰고, 심사 기준·자동 채점 기준을 루브릭 언어로 정리하면 일관성이 생깁니다.[web:23]

## 2. 에듀테크·디지털 러닝 도메인

### 에듀테크(Educational technology, EdTech)
- 교육 목적의 소프트웨어·플랫폼·하드웨어 등 기술을 활용한 교육 공학·산업 전반을 가리킵니다.[web:28]
- 이 공모전의 결과물은 대부분 에듀테크 제품으로 간주되므로, 교육 효과와 비즈니스 모델을 함께 설명해야 합니다.[web:28]

### LMS(Learning Management System)
- 온라인 강좌·과정의 생성, 제공, 학습자 관리와 진행 상황 추적을 위한 소프트웨어 시스템입니다.[web:22][web:25]
- "우리 서비스가 LMS 자체인지, 아니면 기존 LMS 위에 얹는 플러그인인지"를 명확히 하면 구조 설명이 쉬워집니다.[web:22]

### LXP(Learning Experience Platform)
- AI 추천과 콘텐츠 큐레이션, 소셜 기능을 통해 학습자 주도 경험을 강조하는 플랫폼 유형입니다.[web:33][web:45]
- 개인화 피드, 추천, 사용자 생성 콘텐츠가 핵심이면 "LXP 지향" 제품이라는 포지셔닝을 사용할 수 있습니다.[web:36][web:45]

### MOOC·SPOC
- MOOC는 누구나 들을 수 있는 대규모 공개 온라인 강좌, SPOC는 소규모·제한된 대상에게 제공되는 온라인 강좌를 뜻합니다.[web:7]
- 오픈 코스 형태인지, 특정 학교·기관용 폐쇄 코스인지에 따라 운영·비즈니스 모델이 달라집니다.[web:7]

### 블렌디드·플립드 러닝(Blended/Flipped learning)
- 블렌디드는 온라인과 오프라인 학습을 섞는 방식, 플립드는 강의는 집에서 보고 교실에서는 활동·실습을 하는 구조입니다.[web:7]
- 오프라인 코딩 동아리·부트캠프와 연계하는 시나리오를 설계할 때 어떤 비율로 섞을지 논의할 수 있습니다.[web:7]

### 마이크로·모바일 러닝(Micro/Mobile learning)
- 마이크로 러닝은 5~10분 단위처럼 짧게 쪼갠 학습, 모바일 러닝은 스마트폰·태블릿을 중심으로 언제 어디서나 학습하는 방식을 말합니다.[web:4]
- "하루 10분, 한 문제" 같은 컨셉은 마이크로·모바일 러닝 전략과 잘 맞습니다.[web:4]

### 적응형 학습·개인화 학습(Adaptive/Personalized learning)
- 적응형 학습은 학습 데이터에 따라 난이도·경로를 자동 조정하고, 개인화 학습은 개개인의 목표·선호에 맞는 콘텐츠와 피드백을 제공하는 접근입니다.[web:4][web:13]
- LLM 기반 추천·난이도 조절 알고리즘은 이 용어들과 직접 연결해 설명할 수 있습니다.[web:13]

### 게임화(Gamification)·시리어스 게임(Serious games)
- 게임의 점수·레벨·뱃지·퀘스트 요소를 비게임 문맥에 적용하는 것이 게임화이며, 시리어스 게임은 교육·훈련 같은 실질 목적을 가진 게임입니다.[web:5]
- 단순 재미 요소를 넘어서, 어떤 학습 목표를 어떤 게임 메커닉과 연결할지까지 설계하면 좋습니다.[web:5]

### OER(Open Educational Resources)
- 자유롭게 사용·수정·재배포할 수 있는 라이선스로 공개된 교육 자료를 의미합니다.[web:10]
- 예제 문제·콘텐츠를 만들 때 OER를 활용하면 저작권 리스크를 줄이고 개발 속도를 높일 수 있습니다.[web:10]

### 참여·유지(Engagement·Retention)
- 참여는 학습 과정에 얼마나 적극적으로 상호작용하는지, 유지는 플랫폼에 계속 돌아오는 비율을 의미합니다.[web:7]
- UI/알림/보상 구조를 설계할 때 "참여·유지"를 핵심 KPI로 두고 실험 설계를 할 수 있습니다.[web:7]

## 3. 인공지능·생성형 AI 도메인

### 머신러닝(Machine learning)
- 데이터를 통해 패턴을 학습하여 명시적 규칙 없이 예측·판단을 수행하는 인공지능 기법입니다.[web:35][web:41]
- 추천, 난이도 추정, 학습 패턴 분석 등 대부분의 "학습 데이터 기반 기능"이 여기에 속합니다.[web:41]

### 지도·비지도·강화 학습
- 지도 학습은 정답이 있는 레이블 데이터를 활용해 예측 모델을 학습하고, 비지도 학습은 레이블 없이 패턴·군집을 찾으며, 강화 학습은 보상 신호를 통해 환경과 상호작용하며 정책을 학습합니다.[web:35][web:41]
- 난이도 예측은 지도 학습, 유사 학습자 군집은 비지도 학습, 문제 추천 정책 탐색은 강화 학습 예시로 들 수 있습니다.[web:35]

### 딥러닝·신경망(Deep learning/Neural network)
- 인간 뇌 구조를 모사한 다층 인공 신경망을 이용해 이미지·자연어 등 복잡한 패턴을 학습하는 머신러닝의 한 분야입니다.[web:35]
- 코드 자동완성·자연어 튜터는 대부분 대규모 딥러닝 기반 모델 위에서 동작합니다.[web:35]

### 대규모 언어 모델(LLM)
- 대규모 텍스트 데이터로 학습된 신경망으로, 문장 생성·요약·번역·코드 생성 등을 수행할 수 있는 모델입니다.[web:41]
- Cursor, Claude Code, GitHub Copilot 등은 LLM을 활용한 코딩 어시스턴트의 대표 사례입니다.[web:41]

### 파인튜닝(Fine‑tuning)
- 사전 학습된 모델을 특정 도메인 데이터로 추가 학습해 그 분야에 특화시키는 방법입니다.[web:35]
- 공모전에서는 완전한 파인튜닝 대신 프롬프트·RAG 설계로 도메인 특화를 구현하는 전략도 고려할 수 있습니다.[web:35]

### 프롬프트 엔지니어링·시스템 프롬프트
- 모델이 원하는 응답을 내도록 지시문·맥락을 설계하는 것을 프롬프트 엔지니어링이라 하고, 시스템 프롬프트는 모델의 역할·톤·제약을 정의하는 상위 지시입니다.[web:41]
- "튜터로서 행동하라" 같은 시스템 프롬프트와, 문제·힌트를 포함한 사용자 프롬프트를 구분해 설계하면 일관된 응답을 끌어낼 수 있습니다.[web:41]

### 생성형 AI(Generative AI)
- 텍스트, 코드, 이미지, 음성 등 새로운 데이터를 만들어내는 AI 모델 계열을 말합니다.[web:41]
- 설명 자동 생성, 예제 문제 생성, 코드 템플릿 생성 등은 생성형 AI를 교육에 적용한 전형적 사례입니다.[web:41]

### 지능형 튜터링 시스템(ITS)
- 학습자의 상태를 추적하고, 맞춤형 문제·피드백·힌트를 제공해 인간 튜터처럼 작동하는 컴퓨터 기반 튜터 시스템입니다.[web:1]
- LLM 기반 챗봇을 ITS 구조(학생 모델·도메인 모델·튜터 모델)와 연결해 설명하면 연구 맥락이 생깁니다.[web:1]

### 코드 자동완성·코드 리뷰·리팩토링
- 코드 자동완성은 다음 토큰·블록을 예측해 제안하고, 코드 리뷰는 버그·스타일을 검토하는 작업, 리팩토링은 기능은 유지하면서 구조를 개선하는 활동입니다.[web:35]
- 이 기능들을 학습자 수준에 맞게 제한·강화하여 "교육용 코딩 어시스턴트"로 재해석할 수 있습니다.[web:35]

### 컨텍스트 윈도우·토큰·임베딩
- 컨텍스트 윈도우는 모델이 한 번에 고려할 수 있는 입력 길이, 토큰은 모델이 처리하는 최소 단위, 임베딩은 텍스트를 벡터 공간으로 변환한 표현입니다.[web:35]
- 어떤 정보까지 대화에 유지할지, 과거 시도를 어떻게 요약·저장할지 설계할 때 이 제약을 고려해야 합니다.[web:35]

### 환각(Hallucination)·편향(Bias)·안전장치(Safety)
- 환각은 사실이 아닌 내용을 그럴듯하게 생성하는 현상이고, 편향은 특정 집단·관점을 체계적으로 유리·불리하게 만드는 왜곡이며, 안전장치는 이런 위험을 줄이기 위한 필터·가이드라인을 말합니다.[web:34][web:40]
- 정답 검증, 출처 표시, 민감한 조언 차단 설계를 설명할 때 필수적으로 언급해야 하는 키워드입니다.[web:40]

## 4. 소프트웨어 공학·웹 서비스 도메인

> 이 섹션은 일반 소프트웨어 공학 상식에 기반한 요약으로, 구체 정의는 팀이 사용하는 기술 스택 문서와 함께 갱신해 가면 좋습니다.[web:36]

### 프론트엔드·백엔드·풀스택
- 프론트엔드는 사용자가 직접 보는 화면·클라이언트 코드를, 백엔드는 서버·비즈니스 로직·데이터 처리를, 풀스택은 둘 다 다루는 역할을 말합니다.[web:36]

### 아키텍처(Architecture)
- 시스템을 구성하는 모듈과 그 연결 방식을 설계한 구조로, 클라이언트–서버, 마이크로서비스, 서버리스 등 여러 스타일이 있습니다.[web:36]

### API·REST·Webhook
- API는 프로그램끼리 통신하는 인터페이스, REST는 HTTP 기반 자원 지향 API 스타일, Webhook은 이벤트가 발생했을 때 다른 서비스로 콜백을 보내는 방식입니다.[web:36]

### 데이터베이스·스키마·ORM
- 데이터베이스는 구조화된 데이터를 저장·조회하는 시스템, 스키마는 테이블·필드 구조 정의, ORM은 객체와 테이블 매핑을 자동 처리하는 라이브러리입니다.[web:36]

### 클라우드·PaaS·서버리스
- 클라우드는 원격 인프라를 서비스로 사용하는 개념이고, PaaS는 애플리케이션 실행 환경을 제공하는 플랫폼, 서버리스는 서버 관리 없이 함수 단위로 코드를 실행하는 모델입니다.[web:36]

### 버전 관리·Git·GitHub·브랜치·Pull Request
- 버전 관리는 코드 변경 이력을 관리하는 시스템이며, Git은 분산 버전 관리 도구, GitHub는 Git 저장소 호스팅 서비스, 브랜치는 독립 작업 라인, PR은 변경을 메인 브랜치에 합치기 전 리뷰 요청입니다.[web:36]

### CI/CD·배포·모니터링
- CI/CD는 빌드·테스트·배포를 자동화하는 파이프라인, 배포는 새 버전을 사용자 환경에 반영하는 과정, 모니터링은 로그·지표로 상태를 관찰하는 활동입니다.[web:36]

### 테스트·단위 테스트·통합 테스트
- 테스트는 소프트웨어가 의도대로 동작하는지 검증하는 활동으로, 단위 테스트는 함수·모듈 단위, 통합 테스트는 여러 컴포넌트가 함께 동작하는 경로를 검증합니다.[web:36]

### 보안·인증·인가
- 보안은 시스템을 공격·오용으로부터 보호하는 전체 활동, 인증은 "누구인지" 확인, 인가는 "무엇을 할 수 있는지" 권한을 확인하는 절차입니다.[web:36]

## 5. 데이터·평가·러닝 애널리틱스 도메인

### 학습 분석(Learning analytics)
- 학습자와 학습 환경에서 나오는 데이터를 수집·분석해 학습을 이해하고 최적화하려는 연구·실무 분야입니다.[web:1][web:13]
- 어떤 로그를 모으고, 그걸로 무엇을 개선할지 설명할 때 학습 분석 개념을 근거로 사용할 수 있습니다.[web:4]

### 교육 데이터 마이닝(Educational data mining)
- 교육 환경 데이터에서 패턴을 찾는 데이터 마이닝 기법들을 말하며, 학습 분석과 밀접히 연결된 분야입니다.[web:1]

### 로그·이벤트·세션·클릭스트림
- 로그는 시스템 활동 기록, 이벤트는 특정 행동 발생 기록, 세션은 한 번 접속해서 활동하는 구간, 클릭스트림은 페이지·버튼 클릭 흐름 데이터를 의미합니다.[web:1]

### KPI(Key Performance Indicator)
- 조직·서비스 목표 달성 정도를 가늠하는 핵심 지표를 말합니다.[web:10]
- "완료율, 재방문율, 정답률" 등을 KPI로 명시하면 제품·교육 효과 논의를 연결하기 쉽습니다.[web:10]

### 코호트 분석(Cohort analysis)
- 특정 시점이나 속성을 기준으로 묶인 사용자 집단의 행동 변화를 시간에 따라 추적하는 분석 방법입니다.[web:10]

### A/B 테스트·실험
- 두 가지 이상 버전을 무작위로 나눠 제공해 어느 쪽이 지표를 더 개선하는지 비교하는 실험 설계입니다.[web:10]

### 정량 지표·정성 데이터
- 정량 지표는 숫자로 표현되는 데이터(완료율, 체류 시간 등)이고, 정성 데이터는 서술·인터뷰·서베이 응답처럼 맥락이 담긴 텍스트·서사 데이터를 뜻합니다.[web:7]

### 설문·인터뷰·피드백 루프
- 설문·인터뷰는 사용자의 경험·의견을 체계적으로 수집하는 방법이며, 피드백 루프는 데이터 수집→해석→제품 개선→다시 수집의 순환 구조입니다.[web:7]

### 추천 시스템(Recommendation system)
- 사용자의 과거 행동·유사 사용자 정보로 다음에 볼 콘텐츠를 자동으로 제안하는 알고리즘·시스템입니다.[web:41]

## 6. UX/UI·학습 경험(LX) 설계 도메인

### UX(User Experience)·UI(User Interface)
- UX는 제품을 사용하는 전체 경험의 품질, UI는 버튼·레이아웃·타이포그래피 등 눈에 보이는 인터페이스 요소를 뜻합니다.[web:27]

### 서비스 디자인(Service design)
- 사람·프로세스·터치포인트 전반을 설계해 서비스 경험을 일관되게 만드는 접근입니다.[web:27]

### 사용자 여정 맵(User journey map)
- 사용자가 목표를 이루기까지 거치는 단계·감정·터치포인트를 시각적으로 표현한 지도입니다.[web:27]

### 페르소나(Persona)
- 대표적인 사용자 유형을 의인화해 이름·배경·목표·고민을 서술한 가상의 인물 프로필입니다.[web:27]

### 온보딩(Onboarding)
- 신규 사용자가 서비스에 적응하도록 처음 경험·튜토리얼·가이드를 설계하는 과정입니다.[web:22]

### 인포메이션 아키텍처(Information architecture)
- 정보 구조·내비게이션·메뉴 체계를 설계해 사용자가 필요한 정보를 쉽게 찾게 하는 작업입니다.[web:27]

### 사용성(Usability)
- 사용자가 목표를 얼마나 효율적·효과적으로·만족스럽게 달성할 수 있는지를 나타내는 품질 속성입니다.[web:27]

### 접근성(Accessibility)
- 장애 여부와 상관없이 누구나 디지털 제품을 인지하고 조작할 수 있도록 설계하는 것을 말하며, UX의 핵심 원칙으로 강조됩니다.[web:21][web:30]

### 마이크로 카피(Microcopy)
- 버튼·에러 메시지·툴팁 등 작은 텍스트 조각으로, 사용자 행동을 부드럽게 안내하는 짧은 문구입니다.[web:24]

### 에러 상태·빈 상태(Error/Empty state)
- 에러 상태는 문제가 발생했을 때 보여주는 화면, 빈 상태는 아직 데이터가 없을 때의 화면으로, 둘 모두 안내·동기부여 메시지가 중요합니다.[web:24]

### 학습 경험(Learning experience, LX)·몰입(Flow)·인지부하 관리
- LX는 단순 UI를 넘어 학습자의 감정·동기·맥락을 포함한 전체 학습 경험이고, 몰입은 난이도와 능력이 균형을 이룰 때 시간 감각을 잊고 빠져드는 상태를 말합니다.[web:14]
- 인지부하 관리는 정보를 적절히 나누고 시각적으로 정리해 학습자의 작업 기억 부담을 줄이는 설계 전략입니다.[web:14]

### 게임화 요소(Level, Badge, Leaderboard, Quest)
- 레벨·뱃지·리더보드는 성취와 경쟁을 시각화하고, 퀘스트는 명확한 목표와 보상을 가진 미션 단위를 제공합니다.[web:5]

## 7. 비즈니스·제품·MVP·해커톤 도메인

### 문제 정의(Problem statement)·페인 포인트(Pain point)
- 문제 정의는 해결하려는 핵심 문제를 짧고 명확하게 표현한 문장이고, 페인 포인트는 사용자가 현재 겪는 불편·좌절 지점을 뜻합니다.[web:6]

### 타깃 세그먼트·퍼소나·가치 제안(Value proposition)
- 타깃 세그먼트는 공략할 사용자 집단, 퍼소나는 그 집단을 대표하는 인물화, 가치 제안은 그들에게 어떤 가치를 제공하는지에 대한 약속입니다.[web:6]

### PMF(Product–Market Fit)
- 제품이 명확한 시장 니즈를 충족해 자발적인 성장과 유지가 나타나는 상태를 말합니다.[web:6]

### 시장·TAM/SAM/SOM
- 시장은 제품이 겨냥하는 전체 영역이며, TAM/SAM/SOM은 전체 시장, 서비스 가능한 시장, 실제 공략 가능한 시장 규모를 단계적으로 나눈 개념입니다.[web:6]

### MVP(Minimum Viable Product)
- 최소한의 기능으로도 사용자에게 가치를 제공하면서, 시장 반응과 가설을 검증할 수 있는 가장 단순한 제품 버전입니다.[web:6][web:12]
- 린 스타트업의 "빌드–측정–학습" 루프를 빠르게 돌리기 위한 도구로 강조됩니다.[web:6][web:12]

### PoC(Proof of Concept)·프로토타입(Prototype)
- PoC는 기술적·개념적 구현 가능성을 검증하는 실험 버전, 프로토타입은 실제 사용 흐름을 시험해 보는 시제품입니다.[web:6]

### 린 스타트업·빌드–측정–학습
- 린 스타트업은 낭비를 줄이고 빠른 실험을 통해 사업 가설을 검증하는 방법론이며, 빌드–측정–학습은 제품을 만들고, 데이터를 측정하고, 그 결과로 방향을 학습·조정하는 반복 사이클입니다.[web:6][web:12]

### 유저 피드백·인터뷰(Problem/Solution interview)
- 문제 인터뷰는 실제로 해결할 만한 문제가 있는지 확인하는 대화, 솔루션 인터뷰는 제안된 해결책이 매력적인지 검증하는 대화입니다.[web:6]

### 유료 전환(Conversion)·LTV·CAC
- 유료 전환율은 무료 사용자 중 유료로 전환하는 비율, LTV는 한 사용자가 평생 만들어낼 예상 수익, CAC는 한 명을 유치하는 데 드는 마케팅·세일즈 비용입니다.[web:6]

### 해커톤(Hackathon)·트랙·심사 기준(Scoring rubric)
- 해커톤은 짧은 기간 동안 팀이 아이디어를 구현·발표하는 이벤트이고, 트랙은 주제별 경쟁 구분, 심사 기준은 평가 항목과 배점을 정리한 표입니다.[web:23]

## 8. 윤리·정책·거버넌스 도메인

### 개인정보 보호·데이터 보호·GDPR
- 개인정보·데이터 보호는 개인을 식별할 수 있는 정보가 무단 수집·이용·유출되지 않도록 관리하는 활동이며, GDPR은 EU의 일반 개인정보보호법입니다.[web:40]

### 국내 개인정보보호법 개념
- 한국에서는 개인정보보호법·정보통신망법 등이 개인정보 수집·이용·보관·파기에 대한 기준을 정하고 있으며, 온라인 교육 서비스도 이에 따라야 합니다.[web:40]

### 알고리즘 편향(Algorithmic bias)·공정성(Fairness)
- 알고리즘 편향은 데이터·모델 탓에 특정 집단이 체계적으로 불리한 결과를 받는 현상이고, 공정성은 이런 차별을 최소화하려는 원칙입니다.[web:37][web:46]

### 투명성(Transparency)·설명 가능성(Explainability)
- 투명성은 AI가 어떻게 동작하는지·어떤 데이터에 기반하는지 이해할 수 있게 정보를 제공하는 것이고, 설명 가능성은 개별 예측 결과에 대한 이유를 해석 가능한 형태로 제시하는 능력입니다.[web:40][web:46]

### 책임 있는 AI(Responsible AI)
- AI를 안전하고 윤리적으로 개발·배포하기 위한 원칙과 실천을 묶은 프레임워크로, 공정성·신뢰성·프라이버시·포용성·투명성·책임성 같은 원칙을 포함합니다.[web:40][web:34]

### 안전(Safety)·가이드라인(Guidelines)
- 안전은 AI가 예측 불가능한 방식으로 사용자에게 피해를 주지 않도록 설계·운영하는 것이며, 가이드라인은 조직·정부 차원에서 정한 원칙·규범 문서입니다.[web:40]

### 학습 데이터·라이선스·저작권
- 학습 데이터의 출처와 라이선스를 명확히 관리하지 않으면 저작권 침해·약관 위반 문제가 발생할 수 있으며, 재사용 조건을 확인하는 것이 필수입니다.[web:40]

### 디지털 격차(Digital divide)·접근성
- 디지털 격차는 기술·인터넷 접근성 차이로 인해 교육·정보 기회가 달라지는 현상을 말하며, 접근성 있는 설계는 이 격차를 줄이는 수단 중 하나입니다.[web:21][web:30]

## 9. 리서치 실행 루틴(요약)

### 1주차: 맵 만들기 + 키워드 확정
- 위 도메인 구조를 팀에 공유하고, 이번 공모전 전략상 가장 중요한 2~3개 도메인에 별표를 칩니다.
- 각 도메인당 10~20개 키워드를 Obsidian/Notion에 페이지로 만들고, 이 문서 내용을 복사·보완해 나갑니다.[web:1]

### 2주차: 키워드 사전 만들기
- 하루 5~10개씩 키워드를 골라 "한 줄 정의 + 한 줄 대회 맥락"을 팀원 각자가 직접 써 보고, 여기 정의를 자기 언어로 덮어쓰는 식으로 연습합니다.[web:1]

### 3주차: 대회 맥락에 재해석
- 공모전 공지와 심사 기준 문장을 다시 읽으며, 발표문·README에 어떤 이론·용어를 끌어다 붙일지 표시합니다.[web:23]

### 마무리: 1분 설명 연습
- 랜덤 키워드 3~5개를 뽑아 "우리 솔루션과 연결해서 1분 설명"을 서로에게 해 보며, 자주 쓰는 문장은 스크립트로 저장합니다.[web:23]
